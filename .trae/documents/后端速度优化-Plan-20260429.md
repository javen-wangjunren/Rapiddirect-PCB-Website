# 后端速度优化规划（Pages / Media）— Plan

## Summary

目标：把后台 `Pages` / `Media` 页面“每次点击 2–3s 才能用”的体验降到可接受范围，并消除 dev 环境脚本 504 与重复请求带来的抖动。

范围基于：[后端速度优化.md](file:///Users/javen/Desktop/Javen%20Project/PCB/%E9%A1%B9%E7%9B%AE%E6%80%9D%E8%80%83/%E9%80%9F%E5%BA%A6%E4%BC%98%E5%8C%96/%E5%90%8E%E7%AB%AF%E9%80%9F%E5%BA%A6%E4%BC%98%E5%8C%96.md)

用户决策（已确认）：
- 需要同时覆盖：本地 dev + 生产环境
- 允许 Supabase 做 migration（推荐路径）
- Media folder 筛选采用：`media_assets.folder_id`（推荐路径）

成功标准（可验收）：
- 生产环境：进入 `/ems/admin/pages/` 与 `/ems/admin/media/` 的首屏可用时间显著下降；Supabase 列表请求 payload 与耗时随数据量增长保持稳定
- dev 环境：不再出现 toolbar 相关脚本 504；Network 请求数量减少且更稳定
- 数据正确性：folder 过滤、unfiled、移动文件、删除文件行为与现有一致

## Proposed Changes（分阶段、决策完备）

### Phase 0 — 基线采样（不改逻辑，只采集）

目的：把 2–3s 拆成 “文档 TTFB” vs “Supabase 请求”。

执行：
- 生产环境与本地各抓一次：
  - 点击 `/ems/admin/pages/`：记录文档请求 TTFB + `rest/v1/pages` 耗时
  - 点击 `/ems/admin/media/`：记录文档请求 TTFB + `rest/v1/media_folders`/`rest/v1/media_assets` 耗时
- 输出到同一份记录（可以追加到后端速度优化.md 的“基线数据”小节）

验收：
- 得到两类耗时占比，为后续优化排序提供依据

### Phase 1 — 生产侧“后端向”快赢（低风险，高收益）

#### 1.1 Pages 列表：改为服务端分页 + 条件下推

目标：避免全量拉取导致数据量增长后变慢。

改动点：
- 查询层：[adminQueries.ts](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/lib/supabase/adminQueries.ts)
  - 新增 `listPagesForAdminPaged(supabase, { page, pageSize, status, query })`
  - 使用 `range(from, to)` 分页
  - statusFilter 使用 `.eq('status', ...)`
  - query 使用 `ilike`（先做前缀/包含匹配；需要索引支撑见 Phase 2）
  - 返回 `{ items, total }`（total 用 `count: 'exact'`，若数据量大可改 `estimated`）
- UI 层：[AdminPagesIndex.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/AdminPagesIndex.tsx)
  - 从“全量加载 + 本地过滤分页”改为“按页请求”
  - Pagination 依赖后端返回 `total`

验收：
- Network：`rest/v1/pages` 返回行数固定 ~pageSize
- 数据量增加后，耗时基本不随 rows 增长

#### 1.2 Media 列表：改为服务端分页 + 精确列选择

目标：减少 payload 与 DB 扫描范围，为后续 folder_id 改造做准备。

改动点：
- 查询层：[adminMedia.ts](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/lib/supabase/adminMedia.ts)
  - `select('*')` 改为显式列（列表只需：id/path/public_url/title/alt/mime_type/size/created_at）
  - 新增 `listMediaAssetsPaged(supabase, { folderId, unfiled, page, pageSize, query })`
  - 使用 `range(from, to)`
  - query（path/title/alt）先做前端过滤或后端 `ilike`，后续用 pg_trgm 加速
- UI 层：[AdminMediaLibrary.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/AdminMediaLibrary.tsx)
  - 以 “page + pageSize + folderFilter + query” 触发请求，分页由后端 total 驱动
  - 保持已做的并行 refresh/防重逻辑（当前已落地）

验收：
- Network：`rest/v1/media_assets` 返回行数固定 ~pageSize
- 首次进入 Media 页的请求时间与返回体稳定

### Phase 2 — Supabase Migration（核心后端优化）

#### 2.1 索引补齐（立即收益）

新增 migration：`supabase/migrations/phaseX_admin_perf_indexes.sql`（命名按现有约定）

内容建议：
- pages：
  - `create index concurrently if not exists pages_updated_at_idx on public.pages (updated_at desc);`
  - `create index concurrently if not exists pages_status_updated_at_idx on public.pages (status, updated_at desc);`
- media_assets：
  - `create index concurrently if not exists media_assets_created_at_idx on public.media_assets (created_at desc);`
  - `create index concurrently if not exists media_assets_updated_at_idx on public.media_assets (updated_at desc);`（可选）

验收：
- 列表排序/过滤的 Supabase 请求耗时下降（尤其数据量增大后）

#### 2.2 media_assets 增加 folder_id（结构性收益）

新增 migration：`supabase/migrations/phaseX_media_folder_id.sql`

步骤：
1) 扩字段
   - `alter table public.media_assets add column if not exists folder_id uuid null references public.media_folders(id) on delete set null;`
2) 回填（基于现有“path 前缀”规则）
   - 约定：folder 的 `path` 是一个前缀（例如 `ems-home`），asset 的 `path` 若以 `folder.path || '/'` 开头，则归属该 folder
   - 回填 SQL（思路）：
     - 取所有 folders，按 `length(path)` desc 排序（确保更深层先匹配）
     - 逐个更新 `media_assets.folder_id`（或一次性用 lateral join 选最长匹配）
3) 建索引
   - `create index concurrently if not exists media_assets_folder_id_created_at_idx on public.media_assets (folder_id, created_at desc);`
   - `create index concurrently if not exists media_assets_folder_id_idx on public.media_assets (folder_id);`
4) 兼容策略（避免未来不一致）
   - 方案 A（应用层写入，推荐）：上传/移动时由应用显式写 `folder_id`
   - 方案 B（DB 触发器自动维护）：根据 path 与 folders 自动推导（更复杂，不建议第一版就上）

对应代码改动（与 DB 方案 A 匹配）：
- 上传：在 [AdminMediaLibrary.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/AdminMediaLibrary.tsx) 上传成功后 upsert 元信息时，携带 `folder_id`
- 移动：
  - move 操作本就知道目标 folder id（`moveTargetId`），更新 row 时同步更新 `folder_id`（Unfiled => null）
  - 对应改 [updateMediaAssetPath](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/lib/supabase/adminMedia.ts#L163-L176) 入参与 update 字段
- 列表过滤：从 `path like` / `not like` 改为：
  - folder：`eq('folder_id', folderId)`
  - unfiled：`is('folder_id', null)`

验收：
- Media 页 folder/unfiled 过滤查询从“字符串模式匹配”转为“等值/IS NULL”，耗时更稳定
- 移动文件后，列表与详情一致（folder_id 正确更新）

#### 2.3 搜索加速（可选，按数据量启用）

如果需要真正的 contains 搜索（path/title/alt）且数据量较大：
- `create extension if not exists pg_trgm;`
- 对 pages 的 `slug/title`、media_assets 的 `path/title/alt` 建 `gin (col gin_trgm_ops)` 索引
- 查询侧用 `ilike '%q%'` 下推

验收：
- 搜索响应时间与数据量增长更线性可控

### Phase 3 — SSR/部署侧优化（生产体验关键）

这部分严格来说是“部署/渲染层”，但对“每次点击 2–3s”可能是最大头。

#### 3.1 admin 列表页 prerender（推荐）

理由：`Pages`/`Media` 页面本身不依赖服务端数据，数据在客户端拉取，因此可静态化降低 TTFB。

改动点：
- 将以下页面从 `prerender = false` 改为 `true`：
  - `src/pages/admin/pages/index.astro`
  - `src/pages/admin/media/index.astro`
  - `src/pages/admin/pages/ems.astro`（固定 slug）
- `edit.astro` 目前依赖 `Astro.url.searchParams`，若要静态化需把 slug 读取移动到 React 端（可选）

验收：
- 生产环境点击菜单的文档请求 TTFB 显著下降

#### 3.2 部署区域对齐（建议）

如果 Supabase RTT 高：
- 将 SSR 服务部署区域与 Supabase project 区域对齐
- 或尽量把 admin 静态化以绕开 SSR

## Verification（必须逐条完成）

### 功能正确性
- Media：
  - 上传到某 folder：列表中出现在该 folder；Unfiled 不出现
  - Move 到另一个 folder：从旧 folder 消失、新 folder 出现；folder_id 更新正确
  - Move 到 Unfiled：folder_id 变 null；Unfiled 列表出现
  - Delete：storage 对象与 media_assets 行都删除
- Pages：
  - 分页切换稳定；搜索/状态筛选结果正确

### 性能验证（对比基线）
- 生产环境：
  - 文档请求 TTFB（/admin/pages/ 与 /admin/media/）
  - `rest/v1/pages`、`rest/v1/media_assets` 耗时与返回体大小
- dev 环境：
  - Network 不再出现 dev toolbar 相关脚本链路与其 504

### 可回滚性
- 索引与新增字段可保留（不影响读写）；若新逻辑出问题，可临时切回旧过滤方式（path like）作为兜底
