EMS Launch Runbook / Reverse Proxy + Worker + Resend

EMS 上线执行顺序总表

这份文档把当前项目的真实结构拆成一条可执行路线：主域仍然是 rapiddirect.com，对外页面路径保持 /ems/*，其中 /ems/pcb-assembly/ 也是主域下的正式页面；项目本体建议先部署到一台独立服务器， 再通过 Cloudflare 做路径分流、缓存、安全与 Worker 接口绑定。

当前推荐结论： 先走 “独立服务器作为 EMS Origin + Cloudflare 负责边缘流量层”， 暂时不要为了低流量而把整个站点直接迁成 Cloudflare Runtime。
为什么不建议现在直接上 Cloudflare 托管 当前前端使用的是 @astrojs/node standalone，直接切到 Cloudflare 不是“换个部署平台”而是一次运行时迁移。
为什么推荐独立服务器 你后续有“独立出去”的明确可能，先让应用在独立 origin 跑稳，后面换域名、拆站、迁主路由都会更自然。
Cloudflare 仍然是必需层 即便应用部署在服务器，Cloudflare 依然负责 /ems/* 分流、/api/contact Worker、Turnstile、安全与缓存。

本次上线的关键判断

Recommended Path Server First

先部署到新的 Node 服务器，让架构独立成型；Cloudflare 负责入口层与安全层。

Current Runtime

前端项目当前是 @astrojs/node，说明它天然更适合先上 Node 服务器，而不是直接跑 Cloudflare Worker Runtime。

Critical External Services
  • Cloudflare: 路径分流、Worker、Turnstile、Cache
  • Supabase: 页面内容、后台、媒体库
  • Resend: 询盘邮件发送

1. 当前架构应该怎么理解

这次最容易绕晕的点，是你对外看到的是同一个主域 rapiddirect.com， 但背后其实至少有两条链路：一条是 /ems/* 页面内容链路，一条是 /api/contact 表单接口链路。把它拆开后，配置顺序就会清楚很多。

对外一个域名，对内多条服务链路
Public Entry

主域入口

用户始终访问主站域名，而不是临时域名。

  • rapiddirect.com/ems/
  • rapiddirect.com/ems/pcb-assembly/
  • rapiddirect.com/ems/login/
Edge Layer

Cloudflare

负责边缘代理、路由分流、安全和缓存，不一定负责承载整个 Node 应用。

  • /ems/* 分流到 EMS origin
  • /api/contact* 绑定 Worker
  • Turnstile、TLS、Redirect、Cache Rules
App Origin

EMS 独立服务器

运行当前 Astro Node 版本站点，承接前台和后台页面。

  • Astro SSR 前台
  • /ems/admin/* 后台
  • Node 进程、Nginx、部署目录
Data Layer

Supabase

页面内容、SEO 数据、媒体库和后台鉴权都在这里。

  • pages / seo_meta / page_content
  • media_assets / media_folders
  • Auth + Storage + RLS
Inquiry Flow

Worker + Resend

询盘不进入主站 PHP 或 Node 后端，而是直接走 Cloudflare Worker。

  • Worker 校验 Turnstile
  • Worker 用 API Key 调 Resend
  • Resend 以企业域名邮箱身份发出通知

2. 部署方案判断

你现在纠结的，本质不是“流量大小”，而是“当前运行时适配成本”和“后续是否要独立出去”。 低流量不一定等于应该直接上 Cloudflare Runtime，尤其是当现有项目本身就是 Node 目标时。

建议先稳，再追求更轻的托管形态

推荐方案：新服务器先上线，Cloudflare 负责入口层

这是和当前项目最贴合的方案。你把 EMS 应用先放到一台独立服务器上，让它自己成为一个完整 origin， 然后由 Cloudflare 在主域层做路径接入。这样既保住主域 URL，又不强迫项目现在立刻做运行时迁移。

  • 和当前 @astrojs/node 配置最匹配，部署动作最直接。
  • 后续如果 EMS 要独立域名、独立站点，迁移成本最低。
  • 主域下仍可保留 /ems/* 的 SEO 规划，不影响用户感知。
  • Cloudflare 继续承担缓存、Worker、Turnstile、安全能力，不会浪费。

备选方案：直接把整个项目上到 Cloudflare

不是不能做，但它不是“简单省服务器”，而是意味着要把当前 Node 部署目标重新适配到 Cloudflare 的运行环境。 如果你现在的目标是尽快上线并把架构理顺，这条路会增加一轮额外不确定性。

  • 需要把 Astro 的部署目标从 Node 重新适配到 Cloudflare 方案。
  • 后台、Supabase、SSR、静态资源、缓存语义都要重新回归测试。
  • 后面当然能做，但更适合放到“站点跑稳之后”的第二阶段优化。
  • 如果现在强行做，真正节省的不是很多，反而可能拖慢首发节奏。

为什么不选 v0 / Pages / 纯静态托管，为什么选独立服务器 + Cloudflare

这不是在否定这些平台本身，而是结合你们当前项目的实际形态做选择。你们现在面对的重点不是“找一个最便宜的托管”， 而是同时守住 /ems/* 主域子路径、SSR 与后台动态内容、可控缓存、以及未来独立出去的迁移空间。

Not Chosen / v0

不选 v0

v0 更适合快速生成页面和 demo，但你们这里已经进入真实业务阶段。主域子路径反向代理、缓存策略、资源路径和动态内容刷新都需要明确可控， 一旦中间再叠一层平台默认行为，排障会很痛苦。

Not Chosen / Pages

不优先选 Pages

你们当前项目使用的是 @astrojs/node，直接改成 Pages 不是“换个发布入口”，而是一次运行时适配。 首发阶段这样做收益有限，却会引入额外测试面。

Not Chosen / Static

不选纯静态托管

这个项目并不只是几张静态页面。它有 Supabase 动态内容、后台登录、媒体库、询盘接口和 React 交互组件， 所以“纯静态发布 + 文件托管”并不能完整覆盖当前运行需求。

Chosen / Server + CF

选择独立服务器 + Cloudflare

这条路径最符合现在的代码结构，也最利于后续演进。服务器负责承载 Node 应用本身，Cloudflare 负责主域接入、缓存、安全、Worker 和 Turnstile， 角色边界清楚，后面要独立成站也最顺。

决策总结： 现在的最佳策略不是“把一切都塞进某个平台求省事”，而是 让应用运行层和边缘流量层分工明确。 也就是：应用先放独立服务器，Cloudflare 继续做主域入口层。 这样既适合当前技术架构，也给未来 EMS 独立出去留下最小迁移成本。

3. 上线执行顺序

下面这条顺序是按“先冻结架构，再准备服务，再做边缘接入，最后切流量”来排的。 这样做的好处是，不会出现域名先改了、邮件还没通、Worker 还没部署这种交叉阻塞。

推荐按顺序推进，不要跳步骤
01

冻结上线方案

先确定这次首发不做全站 Cloudflare Runtime 迁移，而是“独立服务器 + Cloudflare 入口层”。

本步产出
  • 确定 EMS 独立 origin 存在
  • 确定主站仍对外使用 rapiddirect.com/ems/*
  • 确定 /api/contact 仍由 Cloudflare Worker 承担
目的
  • 避免团队一边准备服务器，一边又临时切去 Cloudflare 托管
  • 把后续所有配置都绑定到一个稳定架构假设上
02

准备新服务器与部署底座

先把应用未来的独立落点准备好，哪怕现在还是挂在主域的 /ems/* 下。

需要完成
  • 新服务器开通，安装 Node、Nginx、PM2 或 systemd
  • 准备部署目录、日志目录、回滚策略
  • 确认服务器能稳定跑 Astro Node standalone
目的
  • 让 EMS 应用先成为一个真正可独立运行的 origin
  • 为未来独立域名、拆站、迁移预留空间
03

准备生产数据与第三方服务

把 Supabase、Resend、Turnstile 这些真正会阻塞上线的生产能力先打通。

需要完成
  • Supabase 生产数据、RLS、媒体 bucket 再确认一遍
  • Resend 验证发信域名，创建 Production API Key
  • Turnstile 创建生产 widget，拿到 site key / secret key
目的
  • 避免站点上线后页面能打开，但询盘提交失败
  • 避免后台能登录、页面能编辑，但发信链路未就绪
04

部署应用到新服务器

先在真实服务器把站点跑起来，再谈主域接入，否则 Cloudflare 规则会无源可接。

需要完成
  • 注入前端环境变量并部署构建产物
  • 确保 /ems/、/ems/pcb-assembly/、/ems/login/ 可访问
  • 本机或临时域名先完成一轮 smoke test
目的
  • 在切主域之前先证明应用本身没问题
  • 把排障范围限制在“应用部署层”，不要和 Cloudflare 配置混在一起
05

配置 Cloudflare 主域接入

这一步是把主域上的 /ems/* 正式接到 EMS 应用 origin 上。

需要完成
  • 确认主域 DNS 在 Cloudflare 且流量走代理
  • 配置 /ems/* 分流到 EMS origin 的方案
  • 打开 HTTPS，保持对外 canonical 仍是主域 URL
目的
  • 让外部用户无感知地从主域访问 EMS 子站
  • 延续你现有的 SEO 和 URL 规划，不暴露临时域名
06

部署 Contact Worker 并绑定路由

把询盘接口独立出来，是这次架构里最清晰的一段，不要让它混进主应用 origin。

需要完成
  • 部署 workers/contact-worker
  • 设置 RESEND_API_KEY、TURNSTILE_SECRET_KEY 等 secrets/vars
  • 绑定路由到 rapiddirect.com/api/contact*
目的
  • 让前端默认的 /api/contact 在生产直接可用
  • 保证表单校验、发信、安全逻辑集中在 Worker 内完成
07

配置缓存、跳转与 SEO 边缘规则

这一步决定你上线后是“稳定快速”，还是“改完后台前台不刷新、HTML 被误缓存”。

需要完成
  • /api/* 明确不缓存
  • /ems/_astro/* 长缓存
  • /ems/* HTML 不要错误强缓存
  • HTTPS、主域规范、必要的 301 跳转
目的
  • 保护动态页面正确性
  • 同时把静态资源性能做上去
08

做完整的生产联调

不是只看首页打开，而是把“页面、后台、媒体、表单、邮件、SEO”全部走一遍。

至少验证这些
  • 前台首页与 pcb-assembly 页面内容正确
  • 后台改内容后前台刷新生效
  • 媒体库上传与引用正常
  • 询盘提交成功，邮箱收到通知且 reply-to 正确
目的
  • 确认不是“每个组件各自看起来都正常”，而是整条业务链路真的闭环

4. 配置矩阵

如果你要把这次上线拆给多人并行做，这张矩阵会最有用。它回答的是： 每个平台到底要配什么，以及每项配置是为了什么。

按平台拆任务最不容易漏项
平台 配置项 需要做什么 目的
Cloudflare DNS / Proxied 确认主域在 Cloudflare 且经过代理流量。 让后续分流、缓存、Worker Route、Turnstile 保护都能生效。
Cloudflare /ems/* 分流 将 EMS 子路径接到独立 origin。 保持主域 URL 不变，同时让 EMS 应用独立运行。
Cloudflare SSL / HTTPS 开启严格 HTTPS 模式与必要跳转。 保证主域访问安全且 canonical 统一。
Cloudflare Worker Route 绑定 rapiddirect.com/api/contact* 到 Contact Worker。 让表单接口走同域路径，减少跨域复杂度。
Cloudflare Cache Rules API 不缓存，静态资源长缓存，HTML 不误缓存。 兼顾动态正确性与资源性能。
Resend 验证发信域名 完成域名 DNS 验证，允许以企业邮箱身份发信。 保证询盘通知能以正式域名邮箱发出，而不是测试邮箱。
Resend Production API Key 创建只用于生产 Worker 的发信 API Key。 让 Worker 拿到调用 Resend 发信的程序级权限。
Turnstile 生产 Widget 创建 widget，绑定正式 hostname，拿 site key / secret key。 拦截垃圾询盘，减少机器人请求直接打邮箱。
Server Node 运行环境 准备 Node、Nginx、进程守护与日志。 承载当前 Astro Node 应用，作为 EMS origin。
Server 部署流程 构建、发布、重启、回滚链路打通。 上线和后续迭代都更稳，不依赖手工临时操作。
Supabase 生产数据与权限 检查表、RLS、seed、媒体 bucket、Auth。 保证前台、后台、媒体库都能在生产正常运行。
Application SEO / 路径输出 补 canonical、sitemap、robots、单 H1、自检链接。 防止主域子路径上线后出现 SEO 偏移与抓取噪音。

5. 环境变量与密钥清单

这部分最适合在真正开始配置之前就列好。它不是“临时补一个 key”，而是整套系统要跑起来的基础清单。

先列全，再逐个落位

前端应用

这些值用于 EMS 站点本体，主要影响前台页面和后台对 Supabase 的连接能力。

PUBLIC_SUPABASE_URL PUBLIC_SUPABASE_ANON_KEY PUBLIC_CONTACT_ENDPOINT=/api/contact PUBLIC_TURNSTILE_SITE_KEY PUBLIC_MEDIA_FOLDER_ID_ENABLED

Cloudflare Worker

这些值只应该进入 Worker，不应该暴露给浏览器。

TURNSTILE_SECRET_KEY RESEND_API_KEY FROM_EMAIL TO_EMAIL ALLOWED_ORIGIN=https://rapiddirect.com

6. 最终上线闸门

下面这些项，只要有一项没过，就不建议把真实流量切上去。它们是“能不能稳定对外”的最低门槛。

上线前最后一轮核查

基础可用性

确认主域子路径、后台、媒体和数据闭环都真实可用。

  • /ems/ 与 /ems/pcb-assembly/ 可访问且资源路径正确
  • /ems/login/ 和后台页面可正常登录与保存
  • 后台改内容后，前台刷新能够立即看到更新
  • 媒体库上传、复制链接、页面引用都正常

询盘链路

这部分必须从浏览器真实提交流程验证，不能只看 Worker 部署成功。

  • Turnstile widget 正常显示与校验
  • /api/contact 返回成功响应
  • 收件邮箱能收到 Resend 通知
  • 点击回复时，收件人是客户填写的邮箱，而不是系统地址

边缘与缓存

如果这里配错，站点经常会出现“明明改了内容却不生效”的假象。

  • API 明确不缓存
  • _astro 静态资源长缓存
  • HTML 未被错误设置为 Cache Everything
  • HTTPS 和主域规范跳转符合预期

SEO 与对外输出

因为这是主域子路径站，canonical 和索引规则必须非常干净。

  • canonical 指向 https://rapiddirect.com/ems/*
  • sitemap 只输出正式主域路径
  • robots 策略和临时环境隔离清楚
  • 不暴露测试域名作为正式页面入口
一句话总结： 这次最稳的首发路径，不是“先省掉服务器”，而是 先让 EMS 应用在独立服务器上跑稳，再用 Cloudflare 把它优雅地接进主域。 这样你既能保住现在的 /ems/* SEO 规划，也能给未来独立成站留下最小迁移成本。
