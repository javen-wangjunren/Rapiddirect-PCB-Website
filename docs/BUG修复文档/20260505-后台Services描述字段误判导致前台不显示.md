## 1. 问题概述

在后台 EMS Service（Services/Our Services）模块中：
- `PCB Design` 这种“段落描述”在后台填写并保存后，前台卡片不展示描述。
- `PCB Manufacturing` 这种“列表描述”在前台有内容，但后台编辑器里显示为“暂无条目”，看起来像没有描述。

## 2. 影响范围

- 后台编辑器无法正确展示/编辑 `services.items[].description` 的真实结构
- 保存时可能把 `description` 写成错误数据结构，导致前台渲染为空（即使不崩溃）

## 3. 复现步骤（修复前）

### 复现 A：后台看不到前台已有的 list 描述
1) 登录后台，进入 `/ems/` 对应页面编辑
2) 展开 EMS Service → Items → PCB Manufacturing
3) 观察 Description 区域显示“暂无条目”，但前台卡片其实有 bullet list

### 复现 B：后台填了描述但前台不显示
1) 登录后台，进入 `/ems/` 对应页面编辑
2) 展开 EMS Service → Items → PCB Design
3) 在 Description 输入一段文字并保存
4) 打开前台 `/ems/`，观察对应卡片描述不显示或变为空

## 4. 根因分析

### 4.1 SchemaForm 的 schema 误判

表单渲染器 [SchemaForm.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/SchemaForm.tsx) 中：
- `isStringArraySchema()` 原本只想识别形如 `{ items: 'array' }` 的“string[] 简写 schema”
- 但它的实现仅判断 `schema.items === 'array'`
- 导致像 `services.items[].description` 这种对象 schema（同时包含 `kind/text/items`）被误判为“数组 schema”，从而：
  - 后台把整个 `description` 当成 string[] 渲染（显示“暂无条目”）
  - 保存时把 `description` 写成 array/string，破坏前台期望的对象结构

### 4.2 错误数据形态会被持久化

由于 `services.items` 是数组，合并逻辑对数组采用“以输入数组为准”，一旦保存了错误结构，会直接覆盖 defaults 对应条目，导致前台描述缺失。

## 5. 修复方案

1) 修复 SchemaForm 误判逻辑：
- 仅把“纯 `{ items: 'array' }` 且无其他键”的对象视为 string[] 简写 schema

2) 针对 `services.items[].description` 提供更明确的编辑 UI：
- `kind` 用下拉（paragraph/list）
- paragraph 显示 Text textarea
- list 显示 Items[] 列表输入

3) 加入兼容旧数据的归一化（load/save）：
- 将历史错误数据（description 为 string / string[] / 异常对象）归一化为 `{kind,text,items}`，避免再保存出错结构
