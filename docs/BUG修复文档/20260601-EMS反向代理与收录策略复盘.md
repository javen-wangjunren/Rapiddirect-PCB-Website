## 1. 一句话结论

这次问题的根因不是“SEO 技术细节”，而是目标选择与落地路径不匹配：我们真正需要的是“子站点对公网不可见”，最简单可靠的方式是 **Cloudflare WAF 锁站**；中途采用 `noindex` 与反代 header 分流的方案，带来了不必要的复杂度，并与 Cloudways Nginx 配置管理的现实情况叠加，最终消耗了大量排查时间。

## 2. 目标与现实

### 目标
- 对外只暴露与收录：`https://rapiddirect.com/ems/`
- 子站点：`pcb.rapiddirect.com` 不希望任何人看到、也不希望被发现/收录

### 误区
- 把“子站不可见”当成“子站可见但 noindex”来处理
- 认为只要在代码里加 meta `noindex` 就能解决“对外不可见”的问题

## 3. 发生了什么（时间线）

### 3.1 约两周前（初始状态）
- 同事在 Cloudways 的 Nginx 上添加了 `/ems/` 的反向代理配置
- 当时反代工作正常：`rapiddirect.com/ems/` 对外可访问

### 3.2 今天上午（配置叠加导致 header 不生效）
- 另一位同事为了让上游能识别真实域名，新增了 proxy headers（例如 `X-Forwarded-Host`）
- 但旧的 `/ems/` location block 没有被移除/更新，导致 Nginx 活跃配置中出现 **重复的 /ems/ 区块**
- 结果：新增 header 没有按预期生效（请求可能仍命中旧 block）

### 3.3 今天下午（删除旧 block 导致反代整体中断）
- 另一位同事移除了旧 `/ems/` block
- 移除后 `/ems/` 反向代理直接返回 404，反代“完全不可用”
- 推断：旧 `/ems/` block 实际上才是承接公网 HTTPS 流量的那个生效配置

## 4. 影响

- `/ems/` 反向代理配置反复修改，出现“看似改了但不生效 / 生效后又 404”的来回
- 团队在 Cloudways 平台侧排障耗时显著（约 4 小时级别）
- 为了达成“子站不收录/不可见”，引入了超出必要范围的工程改动与验证成本

## 5. 根因分析

### 5.1 方案选择偏离真实目标
- **真实目标：不可访问（不可见）**
- **采取手段：noindex + 反代 header 分流（属于“可访问但不收录”）**
- 结果：从“访问控制问题”走成了“SEO 配置问题”，并引入反代 header 依赖

### 5.2 Cloudways Nginx 变更缺少“单一事实源”
- 反代配置不在仓库内版本化管理，依赖人工在平台编辑
- 出现重复 location block 时，缺少快速确认“哪个 block 在承接真实流量”的手段与流程
- 变更没有强制的“变更前导出/备份 + 变更后验收”闭环

### 5.3 缺少最小化验证步骤
- 修改反代/缓存/SEO 任一环节后，缺少统一的最小验收集（HTTP 状态码、关键响应头、反代回源是否正常）

## 6. 最终采用的修复策略（建议的正确方向）

### 6.1 子站不可见：WAF 锁站（访问控制）
- 子站域名开启 Proxied（橙云）后，Cloudflare WAF 才能真正生效
- WAF 规则实现：仅放行 Cloudways 反代出口 IP，其余全部 Block
- 验证方式：`curl -I http://pcb.rapiddirect.com/ems/` 返回 403 且带 `server: cloudflare`

### 6.2 主站可收录：不依赖子站 noindex（SEO 仅对主站）
- 对外收录与访问只围绕 `rapiddirect.com/ems/` 做规范化
- 子站若被 WAF 锁死，则无需再用 `noindex` 增加复杂度（可作为兜底，但不应成为主方案）

## 7. 经验教训（可复用）

### 7.1 先定义“访问控制”还是“SEO 控制”
- 想“外人看不到”：优先做访问控制（WAF/源站 ACL/鉴权）
- 想“能访问但不收录”：再考虑 robots/noindex/canonical

### 7.2 平台配置必须有变更流程
- Cloudways/Nginx 配置变更前：导出/截图/备份现有配置
- 变更中：避免重复 location block；明确唯一入口
- 变更后：跑一组最小验收（见下）

## 8. 最小验收清单（建议固化为 SOP）

### 8.1 子站锁站验收
- `curl -I http://pcb.rapiddirect.com/ems/` 应为 403
- 响应头应包含：`server: cloudflare`、`cf-ray`
- Cloudflare Security Events 应看到命中 Block 规则

### 8.2 主站反代验收
- `curl -I https://rapiddirect.com/ems/` 应为 200/301（按规范）
- 页面可正常加载（浏览器 Network 中主文档不是 403/404）

### 8.3 缓存规则验收（如涉及）
- `/api/*`：`Cache-Control: no-store`
- `/ems/_astro/*`：长缓存（immutable）
- `/ems/*` HTML：避免错误强缓存
