问题 1：网站上线后没有连上 Supabase

根因是服务器里没有正确让前端读到环境变量，导致前端拿不到 Supabase 地址和 anon key。

问题 2：询盘表单一直 404，无法提交

根因不是组件坏了，而是请求没有命中 Cloudflare Worker，或者后面又卡在 CORS / Turnstile。

最终结果： 现在 pcb.rapiddirect.com/ems/ 已经可以正常提交询盘， Turnstile 正常、Worker 正常、Resend 正常，邮箱也能收到邮件。

项目里这几块分别是谁负责的

模块 作用 今天排查时最关键的点
前端站点 apps/ems-site 页面展示、表单 UI、Turnstile 组件、提交请求 要确认 PUBLIC_* 环境变量是否真的进了 build
Cloudflare Worker workers/contact-worker 接收表单、校验 Turnstile、调用 Resend 发信 要确认 route、生效配置、CORS、ALLOWED_ORIGIN

Resend 发询盘邮件 域名验证、DKIM、SPF、Worker secret 都要对

问题 1：网站上线后没有连上 Supabase

现象： 网站虽然能打开，但后台或数据读取不正常，说明前端没有真正连到生产 Supabase。

为什么会这样

代码仓库里的前端会读取环境变量，但服务器本机如果没有正确的 .env，或者改完后没有重新构建，前端 build 里就不会带上这些值。

最终确认的前端环境变量

  • PUBLIC_SUPABASE_URL
  • PUBLIC_SUPABASE_ANON_KEY
  • PUBLIC_CONTACT_ENDPOINT
  • PUBLIC_TURNSTILE_SITE_KEY
  • PUBLIC_MEDIA_FOLDER_ID_ENABLED=false

做了什么

  1. 确认服务器上的 apps/ems-site/.env 真实存在。
  2. 把正确的 Supabase、Turnstile、Contact endpoint 配进去。
  3. 重新执行 build。
  4. 重启网站进程。
结论： 前端环境变量已经生效，网站现在能连到 Supabase。

问题 2：询盘表单一直 404

最开始的现象： 点击提交后返回的是一整页 Astro 404 HTML，而不是表单接口响应。

这个点很重要：当时不是表单组件本身坏了，而是请求没有走到正确的接口。

第一层排查：是不是前端请求地址不对

代码里表单提交依赖 PUBLIC_CONTACT_ENDPOINT。

开始时它是 /api/contact，这在最终主域同源场景下是对的，但在当前临时子站测试场景下会引出更多问题。

第二层排查：是不是没有命中 Worker

后来确认当时访问的页面是：

http://pcb.rapiddirect.com/ems/

而 Worker 一开始只绑了：

rapiddirect.com/api/contact*

所以子域名测试时，请求根本没有走到那个 route。

第三层排查：Cloudflare route 为什么还是不工作

后来又发现 pcb 这条 DNS 记录没有持续走 Proxied，而是为了避开 521 临时用了直连源站。

只要不是 Proxied，Cloudflare Worker route 就不会生效。

最终为了先测试跑通，采用了过渡方案

前端不再临时提交到 /api/contact，而是直接提交到：

https://pcp-contact.leon-a02.workers.dev

这样就先绕开了当前子站未完全稳定的反向代理链路。

我们到底改了哪些内容

位置 修改内容 作用
服务器前端 apps/ems-site/.env 补上 Supabase、Turnstile、Contact endpoint 等变量 让网站 build 时真正吃到生产配置
前端 PUBLIC_CONTACT_ENDPOINT 临时改成 https://pcp-contact.leon-a02.workers.dev 先绕开当前子站未完全稳定的同源 route
Cloudflare Worker route 新增 pcb.rapiddirect.com/api/contact*，保留 rapiddirect.com/api/contact* 保证主域和子域将来都能接 Worker
Worker 变量 ALLOWED_ORIGIN 从单值改成支持多个 origin 让主域和子域都能通过 CORS
workers/contact-worker/src/index.ts 改成支持多个 origin 的匹配逻辑 避免只允许一个域名，后续切主域时又报错
Turnstile widget hostnames 增加 pcb.rapiddirect.com 和 rapiddirect.com 让测试页和正式页都能生成可验证 token
Cloudflare Worker 后台 同步更新 variables 和 routes 让线上真实运行配置跟本地部署配置一致

为什么后来又遇到 CORS / Failed to fetch

这是因为前端已经开始直打 workers.dev，从浏览器角度看，这已经不是同源请求， 所以 Worker 必须明确返回 Access-Control-Allow-Origin。

当时的根因

ALLOWED_ORIGIN 一开始没有和当前页面来源完全匹配。

最后的修正

把 Worker 调整成允许多个 origin，例如：

  • http://pcb.rapiddirect.com
  • https://pcb.rapiddirect.com
  • https://rapiddirect.com
之后通过 curl 验证，已经能看到 access-control-allow-origin: http://pcb.rapiddirect.com， 说明 CORS 已经打通。

为什么 Turnstile 后来又报 invalid-input-response

这一步不代表 Worker 坏了。相反，它说明请求已经到 Worker 了，只是验证码 token 校验失败。

根因

Turnstile widget 的 hostname 必须包含当前访问页面的域名。

最后的处理

  • 给 widget 增加 pcb.rapiddirect.com
  • 保留 rapiddirect.com
  • 刷新页面，重新生成 token 后再提交

最后邮件已经能收到，但为什么显示“代发”

这个现象目前看是正常的，不代表发信失败。

原因是什么

Resend 默认会用 send.rapiddirect.com 这类 return-path / bounce 地址。 某些收件邮箱会把这种显示成“由某某代发”。

这是不是因为 quote@rapiddirect.com 没注册

不是主要原因。Resend 验证域名后，本来就允许从该域名下任意地址发送。

现在是否需要继续处理

如果邮件已经正常进收件箱、不是垃圾箱，可以先不处理，后面再做投递体验优化。

后续正式上线主域时，建议怎么调整

  1. 确认反向代理已经稳定，让正式用户访问的是 https://rapiddirect.com/ems/。
  2. 前端把 PUBLIC_CONTACT_ENDPOINT 从 workers.dev 改回 /api/contact。
  3. 重新 build 并重启前端站点。
  4. 保留两个 Worker route： rapiddirect.com/api/contact* 和 pcb.rapiddirect.com/api/contact*。
  5. 保留多 origin 配置，主域和子域以后都能继续用。
  6. 正式上线后分别在主域和子域再各做一次真实提单测试。
建议： 最终生产形态更推荐走同源 /api/contact，而不是一直直打 workers.dev。这样会更干净，也少一层 CORS。

以后如果再遇到类似问题，优先按这个顺序检查

  1. 页面是否真的吃到了正确的前端环境变量。
  2. 表单提交地址到底是 /api/contact 还是 workers.dev。
  3. 请求到底有没有命中 Worker，而不是落回 Nginx / Astro。
  4. Cloudflare route 是否存在。
  5. DNS 是否真的在走 Proxied。
  6. ALLOWED_ORIGIN 是否和当前页面 origin 一致。
  7. Turnstile widget hostname 是否包含当前测试域名。
  8. Worker secrets 和 Resend / Turnstile key 是否正确。
