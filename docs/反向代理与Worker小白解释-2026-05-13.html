反向代理与 Worker：小白也能理解的版本

这份文档专门解释两件事：什么叫反向代理，以及 为什么你这个 EMS 网站的表单提交要单独用 Worker。我会尽量不用太硬的技术词，而是把它们讲成“前台、总台、秘书、仓库”这种更好理解的角色关系。

这次你不是只遇到了一个技术词，而是同时碰到了 .htaccess、mod_proxy、Nginx、Cloudflare Origin Rule、Worker 这些不同层的东西。 真正让人晕的，往往不是某个名词本身，而是它们都在负责“转发”，但发生的位置完全不同。
反向代理解决什么 解决的是“用户这个请求，后台到底该送到哪里去处理”。
Worker 解决什么 解决的是“请求到了以后，要怎么安全地校验、发邮件、保护密钥”。
你这次为什么会晕 因为你同时在处理主站、子站、Cloudflare、Cloudways、邮件和表单安全，层次太多了。
一句话总纲 先决定“送到哪”，再决定“怎么处理”

前者是反向代理的问题，后者是 Worker / 后端逻辑的问题。

1. 先讲最白话的版本

如果你现在完全不想看技术词，只想先有一个“脑补画面”，那你可以直接记下面两句话。

先把画面感建立起来，再回头看术语会轻松很多
反向代理是什么 你可以把它想成：用户只看到前台门店，但门店后面真正发货的仓库，不一定就在门店里。 用户以为自己一直在找 rapiddirect.com，但后台总台会把请求悄悄送去真正干活的地方。
Worker 是什么 你可以把它想成门口的一个小秘书 / 小保安。它不负责整个网站页面，但专门负责表单提交、机器人校验、邮件发送这种小而关键的任务。
最核心的一句话： 反向代理解决的是 “请求送到哪里去处理”； Worker 解决的是 “这个请求到了以后，要怎么安全地处理”。

2. 什么叫“反向代理”

“代理”这个词听起来很技术，但换成生活里的场景，其实很简单： 你表面上找的是一个窗口，真正处理你事情的人可能在后面另一个房间。

用户看到的是主入口，后台可以自由安排真正处理请求的地方
用户看到的是：

rapiddirect.com/ems/

反向代理做的事：

收到这个请求后，悄悄把它往后转给真正负责 EMS 页面的网站。

真正干活的地方可能是：

pcb.rapiddirect.com/ems/，或者另一台独立服务器上的应用。

小白记忆法： 反向代理不是“把页面改掉”，也不是“让用户跳去另一个网站”。它更像是 总台把来访的人带到正确的办公室，而来访的人自己未必知道后台换房间了。

3. 为什么要这样做

你这次不是为了炫技才做反向代理，而是因为它对现在这个项目结构非常实用。

统一入口

用户继续访问主站路径，体验更统一，不会觉得跳来跳去。

新项目可独立部署

EMS 应用可以先单独放在自己的服务器上，不用一开始就硬塞进老站。

后续迁移更灵活

以后换服务器时，不一定要改前台路径，只是后台接法变了。

排障更清晰

出问题时可以分层判断：主站、子站、Cloudflare、Worker，到底哪一层有问题。

4. 那为什么表单这里又要单独用 Worker

因为“页面展示”和“安全地发邮件”，其实是两种不同的工作。 页面像前台，发邮件像后台秘书。前台可以展示表单，但不适合直接拿着保险箱钥匙去给你发邮件。

浏览器能展示页面，但不适合安全保存私密密钥

页面负责什么

  • 显示表单长什么样
  • 让用户输入姓名、公司、邮箱、附件
  • 展示提交成功或失败提示

发邮件真正需要做什么

  • 验证是不是机器人
  • 保护 RESEND_API_KEY
  • 检查字段、附件和来源域名
  • 真正调用 Resend 发邮件
重点： 浏览器里的前端页面，不能安全地保存 RESEND_API_KEY、 TURNSTILE_SECRET_KEY 这种私密信息。 所以中间必须有一个后端角色，而你这次让 Worker 来当这个角色，正好很顺手。
第 1 步：用户填写表单

前端页面只是负责把数据收集好。

第 2 步：请求交给 Worker

不是直接去找 Resend，而是先交给 Worker。

第 3 步：Worker 做安全检查

查 Turnstile、查字段、查附件、查来源域名。

第 4 步：Worker 调 Resend

发信密钥保存在 Worker 的 secrets 里，不暴露给浏览器。

5. 你这次试过的 3 种反向代理方式

这块是你今天最容易混乱的地方，因为你试的 3 条路，看起来都在“转发请求”，但它们分别发生在 Apache 规则层、Cloudways 主站服务器层、Cloudflare 边缘层。

同样都是“转发”，但位置不同、能力不同、限制也不同
Attempt 1

.htaccess 方案

你先试的是在 Cloudways 主站应用的 .htaccess 里，用 Apache 的重写规则去代理 /ems/*。

RewriteRule ^ems/?$ https://pcb.rapiddirect.com/ems/ [P,L]
RewriteRule ^ems/(.*)$ https://pcb.rapiddirect.com/ems/$1 [P,L]
  • 白话理解：想让主站的“门卫”在收到 /ems/* 时，把客人送去 pcb.rapiddirect.com/ems/*。
  • 为什么看起来合理：路径关系很直接，语义也简单。
  • 为什么放弃：Cloudways 明确说 reverse proxy via .htaccess is not supported，也就是这条路在它当前环境里不成立。
结果：放弃
Attempt 2

Cloudways 服务器层 Nginx 方案

后来 Cloudways 客服明确告诉你：如果要代理，应该在服务器层的 Nginx 里加 location block，而不是走 .htaccess。

location = /ems {
    return 301 /ems/;
}

location /ems/ {
    proxy_pass https://pcb.rapiddirect.com;
    proxy_http_version 1.1;

    proxy_set_header Host pcb.rapiddirect.com;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_ssl_server_name on;
    proxy_redirect off;
}
  • 白话理解：不是让应用目录里的“小规则文件”处理，而是让主站服务器自己的“大门卫”直接接手这件事。
  • 为什么更靠谱：Nginx 本来就非常适合做路径转发和反向代理。
  • 现在状态：Cloudways 已经帮你开启 mod_proxy，并且正在帮你调整服务器层 Nginx。
结果：等待 Cloudways 调整
Attempt 3

Cloudflare Origin Rule 方案

你也试过让 Cloudflare 在边缘层直接改源站，也就是让用户访问 rapiddirect.com/ems/* 时，Cloudflare 内部改送去别的 origin。

(http.host eq "rapiddirect.com" and starts_with(http.request.uri.path, "/ems"))
  • 白话理解：想让“站外的大门口保安”直接决定把 EMS 请求送去哪台源站。
  • 为什么很诱人：如果能在 Cloudflare 层搞定，主站服务器就少改一层。
  • 为什么放弃：你需要的那种 origin override 能力，在当前套餐下受限，需要 Enterprise 级别。
结果：放弃
你现在这 3 条路的最终结论： .htaccess 这条路在 Cloudways 环境里不支持；Cloudflare Origin Rule 这条路在当前套餐下不现实；所以现在最可行的路，就是 让 Cloudways 在服务器层的 Nginx 去处理 /ems/* -> pcb.rapiddirect.com/ems/*。

6. 这些东西分别在哪一层

小白最容易把这些词混成一锅，是因为它们都在“参与转发”，但其实站在的楼层不一样。你可以把它们想成从外到内的 4 层。

看图先理解：一次请求从外到内会经过哪 4 层

你可以把这套结构想成“用户从街道走进大楼，再被前台带到真正办公室”。用户只看见入口，但后台会一层层把请求送到正确位置。

1

用户浏览器

用户只知道自己访问的是 rapiddirect.com/ems/*，他并不知道后面到底是哪台服务器在干活。

2

Cloudflare

这是站外的总入口，负责 DNS、CDN、SSL、安全和 Worker route。它更像门口总闸，但这次没直接接管 /ems/* 的改源站工作。

3

Cloudways 主站服务器

这里是真正接住主域请求的地方。你现在要在这一层，把 /ems/* 转给真正负责 EMS 内容的源站。

4

EMS 独立站

也就是 pcb.rapiddirect.com/ems/*。这里才是真正返回 EMS 页面内容、后台数据和应用逻辑的地方。

.htaccess

属于主站应用层的小规则文件。你试过让它代理，但 Cloudways 这条路不支持。

mod_proxy

是 Apache 的“具备代理能力”的模块，像是给门卫增加带路能力。

Nginx

是主站服务器层更强的前台总调度。现在最有希望在这一层完成 /ems/* 转发。

Origin Rule

属于 Cloudflare 边缘层的改源站规则。思路对，但当前套餐下关键能力不可用。

Worker

不负责整站页面转发，它更像是表单秘书，专管安全校验、密钥保护和发信。

第 1 层：用户浏览器

用户只知道自己访问的是 rapiddirect.com/ems/。

第 2 层：Cloudflare

这是站外的大门口保安，负责 CDN、SSL、安全、Worker route 等。

第 3 层：Cloudways 主站服务器

这里是主站真正接请求的地方，你现在要在这层做 /ems/* 的代理。

第 4 层：EMS 独立站

也就是 pcb.rapiddirect.com/ems/* 这套应用真正返回内容的地方。

东西 它是什么 你可以把它想成 它这次负责什么
.htaccess Apache 应用层规则文件 应用目录里的小告示牌 你试过让它代理 /ems/*，但 Cloudways 不支持这条路
mod_proxy Apache 的代理能力模块 让门卫具备“带路”的能力 客服帮你开了，但最终仍建议走 Nginx 层，而不是 .htaccess 层
Nginx Web Server / 反向代理层 真正的大门卫 现在最有希望承接主站 /ems/* 转发
Origin Rule Cloudflare 边缘层改源站规则 站外入口保安的调度规则 思路成立，但当前套餐不现实
Worker Cloudflare 的轻量后端逻辑 专管表单的小秘书 验证 Turnstile、保护 secret、调用 Resend 发邮件

7. 最后你应该怎么理解整套结构

你现在可以把整套东西压缩成一条非常清楚的链路。只要记住这条链路，很多零散名词就不会再那么吓人。

一张图看懂：现在整套请求最终会怎么流动

这张图最适合你以后拿来复盘。只要记住“页面请求走主站反代，表单请求走 Worker”，整个架构就不会再乱。

用户访问

rapiddirect.com/ems/*

→

Cloudways 主站反向代理

主站接住请求后，把它送去真正负责 EMS 页面内容的源站。

→

EMS 独立站

pcb.rapiddirect.com/ems/* 返回页面、内容和后台数据。

→

用户看到页面

地址栏始终还是主站路径，看起来像同一个网站。

页面链路

主站反向代理主要负责“把页面请求送到正确的站点”，也就是解决“送到哪”。

表单链路

用户提交表单后，请求不直接找 Resend，而是单独去 Worker，由 Worker 校验 Turnstile、保护 secret，再调用 Resend 发邮件。

你可以把它记成一句话： 页面访问这件事，走的是 主站反向代理； 表单发信这件事，走的是 Worker + Resend。 前者负责“找对办公室”，后者负责“安全地办事”。
用户看到：

rapiddirect.com/ems/*

主站反向代理负责：

把这些页面请求转交给真正负责 EMS 页面内容的 pcb.rapiddirect.com/ems/*

EMS 站点负责：

显示页面、内容、后台数据

Worker 负责：

接表单、查 Turnstile、保护密钥、调用 Resend

Resend 负责：

真正把邮件发出去

最后一句最重要的话：

反向代理让用户始终看到统一入口；Worker让表单发信逻辑既安全又独立。 前者解决“送到哪”，后者解决“怎么处理”。

你现在真正处于哪个阶段： EMS 独立站本身已经跑起来了，Worker 也承担了表单链路。现在真正还差的一步，就是 Cloudways 在主站服务器层把 /ems/* 的反向代理配置补好。 这一步一旦完成，整套链路就会更像正式生产形态。
