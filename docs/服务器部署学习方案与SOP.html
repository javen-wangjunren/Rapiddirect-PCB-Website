服务器部署学习方案与 SOP

这份文档是写给“完全不懂服务器和代码的人”的。目标不是把你训练成运维工程师，而是帮你建立一张 能看懂、能复盘、能照着做的地图。你昨天不是“学不会”，而是一下子同时接触了 命令行、Linux 用户权限、Node 应用、Nginx、Cloudflare、反向代理、GitHub 发布这几门课。

请先记住一句话： 你真正需要掌握的，不是所有底层原理，而是把复杂流程变成 固定模板 + 固定命令 + 固定判断顺序。
你的身份 SEO 从业者，不需要成为全栈或运维，只需要做到“会部署、会更新、会判断问题在哪一层”。
你的真实目标 以后拿到一个 Astro 网站，不再慌，知道该用什么方式部署、该在什么目录执行什么命令。
这份文档的角色 把昨天那一整天的碎片操作，整理成一套能复用的知识大纲和部署 SOP。

先看结论

Learning Goal 够用即可

重点不是学很深，而是学到“以后能部署和更新任何一个类似的网站”。

你的 4 个知识面
  • 命令行工具
  • 服务器基础与权限
  • 反向代理与端口
  • GitHub 与发布流程
最重要的心态

以后不要把“命令”当成记忆负担，而要把它们理解成一组固定动作：登录、进目录、拉代码、构建、重启、验证。

1. 先看全景地图

你昨天真正碰到的，不是一个“服务器命令问题”，而是一条完整的网站链路。只要你先看懂这条链路，后面每个名词都会有位置感。

先看地图，再学每个站点
Local

本地电脑

你平时写代码、改文件、测试页面的地方。可以把它理解成“设计室和编辑室”。

GitHub

代码仓库

相当于“云端总文件柜”。以后标准做法是：本地改完，先推 GitHub，再让服务器拉更新。

DigitalOcean

EMS 独立服务器

真正运行 Astro/Node 应用的电脑。这里有 Ubuntu、Node、PM2、Nginx 和你的项目代码。

Cloudways

主站服务器

继续负责主站。你最后选择让它把 /ems/* 反向代理到 EMS 独立服务器。

Cloudflare

大门口和保安

主要负责 DNS、CDN、SSL 和流量入口。原本尝试让它直接改源站，但 Enterprise 功能限制了这条路。

Nginx

前台接待员

外部请求先打到 Nginx，再由它转给内部真正干活的 Node 应用。

Node / Astro

真正出页面的人

Astro 源码会被 build 成 Node 可运行产物，最终由 Node 进程输出网页内容。

PM2

值班经理

负责盯着 Node 进程，让网站常驻运行，重启服务器后也能自动再拉起。

最重要的链路一句话： 用户访问主站路径 rapiddirect.com/ems/*，主站 Cloudways 收到后再把这段请求代理到 pcb.rapiddirect.com/ems/*，而这个子域名背后就是你在 DigitalOcean 上跑起来的 Astro/Node 网站。

2. 昨天其实发生了什么

你会晕，不是因为你笨，而是昨天连续完成了“选方案、买服务器、配权限、拉代码、跑 Node、配 Nginx、试反向代理、改主站方案”整套流程。

这不是一天一课，这是一天八课
01

放弃 Cloudways 直接跑 Node

你先确认了 Cloudways 对 Node 不是原生支持，而是借 PHP 容器和代理绕过去。

  • 你判断这条路对未来独立站不够干净。
  • 所以决定单独买一台 DigitalOcean 服务器跑 EMS。
02

选服务器配置

你选择了 Ubuntu、Basic、合适的 CPU/RAM，而不是一上来安装面板或加各种托管能力。

  • 这是在选“新办公室的大小和配置”。
  • 重点是能稳定跑 Node 网站，而不是一开始堆很多附加功能。
03

学会 SSH 登录

你开始接触 SSH、公钥、私钥、用户、密码登录等概念。

  • SSH 就是“远程遥控器”。
  • 公钥像门锁认可的钥匙信息，私钥像你自己手上的钥匙。
04

做服务器初始化

你创建了日常运维用户，更新系统，配置基础安全和依赖。

  • 这是在装修办公室：先通电、配门禁、分工位。
  • 其中 javen 是日常用户，root 是超级管理员。
05

安装 Node、pnpm、PM2

你让这台服务器具备了运行 Astro/Node 网站的能力。

  • Node 是运行网站程序的引擎。
  • PM2 是让网站程序一直在线的值班经理。
06

从 GitHub 拉代码

你不是用 SFTP 作为主流程，而是把 GitHub 作为主发布入口。

  • 这是对的，因为以后版本能追踪、能回滚。
  • 服务器只是运行环境，不应该成为主编辑器。
07

Build 并启动 Astro 应用

你在正确的应用目录 apps/ems-site 里完成了构建和启动。

  • pnpm build 把源码做成产物。
  • dist/server/entry.mjs 是最终运行入口。
08

用 Nginx 把网站对外开放

你把外部访问的 80 端口转发到了内部 Node 服务端口。

  • 没有 Nginx，Node 程序只是“在办公室里工作”。
  • 有了 Nginx，外部用户才能正常访问页面。
09

试图用 Cloudflare Origin Rules

你尝试让 Cloudflare 把 rapiddirect.com/ems/* 指到新的源站。

  • 思路没错。
  • 但真正需要的 DNS/Host override 在你当前套餐下不可用。
10

试图在主站 .htaccess 里做代理

你试了 Apache 的 [P] 代理规则，但主站环境返回了 500。

  • 这一步暴露出 Cloudways 当前栈对 .htaccess 反向代理不支持。
  • 不是思路错，是实现方式不被当前环境接受。
11

切回主站 Nginx 级别反代

Cloudways 客服确认，要由他们在主站 Nginx 层加 location block 来完成代理。

  • 这就是你现在距离“全链路跑通”最后的一步。
  • 本质上是让主站把 /ems/* 转给新服务器。

3. 名词翻译：把专业词讲成人话

下面这些词，你不用追求“标准教材定义”，先追求“脑子里能有画面”。只要有画面，后面做事就不会慌。

先看懂角色，再记名词
SSH

SSH

远程登录服务器的方式。你坐在自己电脑前，却能像坐在服务器面前一样发命令。

形象理解：SSH 就像一把“远程遥控器”。
Key Pair

公钥 / 私钥

服务器记住公钥，本地保管私钥。公钥负责“允许谁进门”，私钥负责“真正开门”。

形象理解：公钥像门禁系统里存的指纹模板，私钥像你自己的手指。
Users

root / javen

root 是超级管理员，什么都能改；javen 是日常运维用户，更安全。

形象理解：root 是大楼总控钥匙，javen 是日常办公卡。
Node.js

Node.js

让服务器能运行 JavaScript 程序的运行环境。Astro build 之后生成的服务端产物，要靠它执行。

形象理解：Node 就像厨房的炉灶，没有它厨师有菜谱也做不了菜。
pnpm / npm

npm / pnpm

安装项目依赖的工具。网站不是只靠你自己的代码，还需要 Astro、React 这些外部包。

形象理解：它们像采购系统，负责把做菜需要的原材料搬进仓库。
PM2

PM2

负责管理 Node 进程的工具。它会让网站常驻运行，进程挂了可以重启，机器重启后还能自动恢复。

形象理解：PM2 是值班经理，盯着员工别下班溜走。
Nginx

Nginx

对外接待流量的 Web Server。外部请求先到它，再由它转给内部真正跑网站的 Node 程序。

形象理解：Nginx 是前台接待员，不自己做页面，但负责把顾客带到正确窗口。
Port

端口

一个程序在服务器上的门牌号。80 是 HTTP，443 是 HTTPS，4321 是你这次 Astro 程序的内部门牌号。

形象理解：一栋楼里很多房间，每个房间门牌号不同，端口就是房间号。
Reverse Proxy

反向代理

外面的人以为自己在访问 A，其实后台被转到 B。地址栏不变，但服务来源换了。

形象理解：顾客在前台点餐，前台把订单送到另一家后厨做，但顾客并不知道。
.htaccess

.htaccess

Apache 的目录级规则文件。你在主站上尝试用它写代理规则，但当前 Cloudways 场景不支持这条实现方式。

形象理解：这是“这层楼自己贴在门口的管理告示”。
mod_proxy

mod_proxy

Apache 的代理能力模块。没有它，.htaccess 里的代理规则写了也没法工作。

形象理解：这是前台接待台的“转单功能插件”。
Origin Rule

Cloudflare Origin Rule

Cloudflare 在边缘层改“请求最终发去哪里”的规则。你本来想用它，但关键 override 受套餐限制。

形象理解：这是门口保安在内部改派送目的地，而不是让大楼前台改。

4. 命令行速查：命令是什么、干嘛用、像什么

你不用把命令都背成考试题，而是记成“我在做什么动作”。以后看到这些命令，脑子里能对应动作就够了。

命令 = 一种固定动作
命令 / 快捷键 它在干什么 形象理解 你这次什么时候用到
pwd 查看当前所在目录 看自己现在站在办公室哪一间 确认是不是站在正确项目目录
ls -la 列出当前目录所有文件 打开抽屉看看里面有什么 确认是否有 package.json、dist、.ssh
cd 进入某个目录 从一个房间走到另一个房间 切到仓库根目录、应用目录
cp 复制文件 复印一份文件 复制 authorized_keys 给新用户
mkdir 创建目录 新建一个文件夹 创建项目目录、.ssh 目录
sudo 临时用管理员身份执行命令 借用总控钥匙办事 改系统配置、装软件、改 Nginx
apt Ubuntu 安装软件的工具 系统软件应用商店 更新系统、安装 Nginx、curl 等
ufw 防火墙管理工具 大楼门禁系统 开放 22 / 80 / 443
git clone / git pull 拉取 GitHub 代码 把云端文件柜里的最新文件拿下来 第一次部署、以后更新代码
nvm 管理 Node 版本 切换不同型号的炉灶 安装和固定 Node 22
npm / pnpm 安装项目依赖 采购做菜原材料 pnpm install、pnpm build
curl 用命令行访问一个网址 不用浏览器，直接敲门看对方有没有回应 测试本机 4321、测试 Nginx 是否代理成功
chmod 改文件权限 规定谁能看、谁能改、谁能进 设置 .ssh 权限
chown 改文件所有者 把某个文件转给某个人负责 把目录和 key 文件归给 javen
Ctrl + C 中断当前命令 喊停 结束前台运行的 Node 进程
Ctrl + O 在 nano 里保存 按下保存按钮 保存 Nginx 配置文件
Ctrl + X 退出 nano 关掉编辑器窗口 退出 Nginx 配置编辑界面

5. 部署 SOP：以后再部署 Astro 网站，按这套来

这部分是你真正能重复使用的。以后不要再把部署想成“现场发挥”，而是当成一套工厂化流程。

SOP = 固定流程，不靠记忆
Decision Tree

先判断网站属于哪一类

  • 独立域名 / 子域名，且没复杂后端：优先 Vercel / Netlify / Cloudflare Pages。
  • 独立域名 / 子域名，但需要 Node SSR：优先 VPS（DigitalOcean 这类）。
  • 挂在主站路径下，比如 example.com/ems/*：复杂度最高，要额外处理反向代理。
Important

主站路径挂载永远更复杂

  • 因为它天然牵涉“谁对外、谁在后台、谁改路径、谁改源站”。
  • 以后只要业务允许，优先考虑子域名，比如 ems.example.com，会简单很多。
SOP 1

服务器初始化

这是以后每次新买服务器都几乎一样的步骤。

  • 买服务器，选 Ubuntu LTS。
  • 配 SSH key，登录 root。
  • 更新系统、装基础工具。
  • 创建日常用户，给 sudo 权限。
  • 开防火墙，只放行 22 / 80 / 443。
  • 安装 Node、pnpm、PM2、Nginx。
SOP 2

代码部署

以后一律以 GitHub 为主，不把 SFTP 当主发布方式。

  • git clone 仓库
  • 进入实际应用目录
  • pnpm install
  • pnpm build
  • pm2 start 或 pm2 restart
SOP 3

对外访问

先把应用跑起来，再让外部世界能访问。

  • Node 应用先在本机端口跑起来，比如 4321。
  • Nginx 把 80/443 转发到内部端口。
  • 先用服务器 IP 或源站子域名测试。
  • 最后再接正式域名和 Cloudflare。
SOP 4

上线前验收

不是“能打开首页”就算上线，而是整条业务链要闭环。

  • 首页和关键子页面能访问
  • 资源不 404
  • 后台登录正常
  • 表单正常
  • 缓存和代理没有把页面搞错
你的标准发布命令块： 
cd /var/www/ems-site/Rapiddirect-PCB-Website
git pull

cd /var/www/ems-site/Rapiddirect-PCB-Website/apps/ems-site
pnpm build
pm2 restart ems-site
如果改了依赖，再多加一步： 
pnpm install

6. 如何简化：以后不想敲这么多命令怎么办

这是你最关心的部分。答案是：同类流程还是会存在，但可以被“标准化、脚本化、半自动化、面板化”。

复杂度不会消失，但可以被封装
Level 1

命令清单化

把常用命令整理成固定文档，部署和更新都照抄，不靠记忆。

cd /var/www/ems-site/Rapiddirect-PCB-Website
git pull
cd apps/ems-site
pnpm build
pm2 restart ems-site
Level 2

写成 deploy.sh 脚本

这是最适合你下一阶段做的。以后只需要执行一条命令。

#!/bin/bash
set -e

cd /var/www/ems-site/Rapiddirect-PCB-Website
git pull

cd /var/www/ems-site/Rapiddirect-PCB-Website/apps/ems-site
pnpm build
pm2 restart ems-site
Level 3

GitHub Actions 自动部署

本地 push 后，GitHub 自动 SSH 到服务器，执行拉代码、构建、重启。

  • 优点：你几乎不用手动登录服务器。
  • 缺点：初次配置比脚本复杂一点。
Level 4

面板化管理

如果以后你会管理多个 Astro/Node 网站，再考虑 Easypanel 或 Coolify 这类更现代的面板。

  • 好处：GitHub 自动部署、应用管理更图形化。
  • 前提：你先把底层流程学明白，不然面板只会把复杂度藏起来。
Recommended

对你来说最合理的简化路线

  • 第一步：固定一套技术栈和目录结构。
  • 第二步：整理标准命令清单。
  • 第三步：做 deploy.sh 脚本。
  • 第四步：以后再考虑 GitHub Actions。
  • 第五步：当站点数量变多，再上现代面板。
Not Yet

现在不建议你立刻做的事

  • 不要立刻学 Docker / Kubernetes。
  • 不要一开始就做很重的自动化。
  • 不要让 SFTP 成为主发布方式。
  • 不要每次都换一套部署架构。

7. 给小白的学习方案：学到什么程度就够

你的目标不是“知道一切”，而是“以后自己能完成部署和更新，不再被术语压住”。所以学习顺序必须有轻重。

先会更新，再会部署，再会排障

阶段 1：先学会更新网站

这是离你的本职工作最近、最有用的一层。

  • 会 SSH 登录
  • 会进入正确目录
  • 会 git pull
  • 会 pnpm build
  • 会 pm2 restart

阶段 2：再学会部署网站

知道一个新的 Astro/Node 网站怎么从 0 跑起来。

  • 理解 Node / PM2 / Nginx 分工
  • 理解端口和反向代理
  • 能照着 SOP 初始化一台服务器

阶段 3：最后学会排障

知道问题大概出在 GitHub、服务器、Nginx、Cloudflare、还是主站代理层。

  • 会看日志
  • 会用 curl 测试链路
  • 会判断 404、500、502 大概属于哪层

阶段 4：再考虑自动化

只有手动流程跑顺了，自动化才会真正帮你，而不是把问题藏起来。

  • 先做脚本
  • 再做 GitHub Actions
  • 最后视站点数量考虑面板
对你这种角色，真正“够用”的目标是： 你不需要变成专业运维，而是要能做到：看懂这条链路、按 SOP 做事、按命令清单发布、知道问题在哪一层。达到这个程度，网站部署这件事就不再是你害怕的领域，而会变成一个可重复执行的流程。
最后送你一句总纲： 以后再碰到新的 Astro 网站，不要先问“这个命令怎么背”，而是先问： 这个网站属于哪一类、代码在哪、谁在运行它、谁在对外接待、谁在做代理。 只要你把这五个问题想清楚，部署就会从“黑箱”变成“拼图”。
