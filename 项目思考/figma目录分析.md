# `.figma` 目录分析与正确使用方式

本文档用于说明当前项目中 `.figma/` 目录的来源、内容结构、可用价值、错误用法，以及它在当前开发流程中的正确位置。

## 1. 结论先说

`.figma/` 目录本质上是 **Figma / TRAE 导出的设计稿产物目录**，它不是正式项目源码目录。

它的主要价值不是“直接拿来开发”，而是：

- 看设计稿结构
- 理解模块顺序与视觉层级
- 提取 PNG / SVG / 背景图等素材
- 辅助判断 spacing、字号、配色、图标、背景等视觉信息

不应该做的事：

- 直接把整页导出的 `index.jsx` 放进正式项目
- 直接把整页导出的 `index.module.scss` 作为生产代码样式体系
- 根据导出代码反向决定字段结构与 schema

## 2. 当前 `.figma/` 目录里有什么

根据当前仓库结构，`.figma/` 里主要有这几类内容：

### 2.1 `.figma/49_156/`

这是某个具体设计稿页面的导出目录，目前你这个 PCB EMS 首页设计稿对应的核心代码导出就在这里。

里面主要有：

- `index.jsx`
- `index.module.scss`

这两个文件通常表示：

- `index.jsx`：整页 React 结构导出
- `index.module.scss`：与这整页结构绑定的样式导出

### 2.2 `.figma/image/`

这是图片与图标资源目录，里面包含：

- `.png`
- `.svg`
- 设计稿截图 `screenshot_49_156.png`

这些文件通常包括：

- 背景图
- 图标
- 装饰图
- 卡片图片
- 设计稿快照

### 2.3 其他辅助目录

例如：

- `.figma/.vscode/`

这类目录通常只是导出环境或编辑器配置辅助，不属于项目核心开发内容。

## 3. 逐类分析：哪些有用，怎么用

### 3.1 `index.jsx` 是什么

`index.jsx` 是 TRAE / Figma 导出的“整页 JSX”。

它的特点通常是：

- DOM 层级很深
- class 名自动生成且语义弱
- 直接引用相对路径图片
- 一次性把整页内容塞进一个组件里
- 更接近“视觉快照代码”，而不是“工程化组件代码”

从你当前的 `index.jsx` 看，已经能明显看到这些特征：

- 一整个页面都在一个 `Component` 中
- 结构是整页平铺，而不是模块化拆分
- 大量 `styles.xxx` 类名与自动命名 wrapper
- 资源用 `../image/*.svg`、`../image/*.png` 相对引用

### 3.2 `index.jsx` 能拿来做什么

可以拿来：

- 看页面大结构
- 找模块顺序
- 提取文案
- 理解图片区块与按钮分布
- 识别哪些内容天然应该归为一个模块

不建议拿来：

- 直接复制到 Astro 项目里作为正式页面
- 基于它直接继续改造成可维护版本

原因：

- 不可维护
- 没有清晰模块边界
- 很难接 schema
- 很难复用
- 后面接 Supabase 时基本会推倒重来

### 3.3 `index.module.scss` 是什么

这是与整页 JSX 紧耦合的样式文件。

从当前文件看，它有明显的导出痕迹：

- 固定宽高，如 `1920px`
- 巨大的整页高度，如 `8700px`
- 大量绝对定位
- 一页式视觉排布
- 直接写背景图 URL

**这类 SCSS 的价值主要是“视觉参考”**，不是“样式体系本身”。

可以拿来：

- 查颜色值
- 查字体大小和字重
- 查边距、圆角、阴影
- 查背景图和图标资源

不建议拿来：

- 直接作为正式项目样式
- 继续在这个文件上扩展需求

### 3.4 `.figma/image/` 是什么

这个目录是 `.figma/` 里最有价值、最可直接使用的部分。

它可以作为：

- 图片素材源
- SVG 图标源
- 背景图源
- 设计稿截图参考

这些资源可以在筛选后迁移到正式项目里，例如：

- 放到 `src/assets/images/`
- 放到 `src/assets/icons/`
- 后续可运营图片迁移到 Supabase Storage

## 4. 哪些内容可以直接用

### 可以直接用

- `screenshot_49_156.png`
- `.figma/image/` 下的 PNG 素材
- `.figma/image/` 下的 SVG 图标

用途：

- 作为视觉参考
- 作为临时开发素材
- 作为 mock 阶段组件图片来源

### 可以参考用

- `index.jsx` 的文案与模块顺序
- `index.module.scss` 的视觉细节

用途：

- 帮助你拆模块
- 帮助你人工决定字段
- 帮助你写组件时对齐设计

### 不应直接用

- 整页 `index.jsx`
- 整页 `index.module.scss`

原因：

- 这不是工程化代码
- 这不是可维护组件系统
- 这不是适合接 schema / Supabase 的结构

## 5. 正确理解“TRAE 导入设计稿”

TRAE 导入设计稿的正确作用，不是帮你生成最终前端代码，而是帮你：

- 快速看到设计结构
- 快速拿到素材
- 快速获取视觉细节

它是一个“辅助输入工具”，不是“最终实现工具”。

### 错误用法

```text
Figma → 导出整页 JSX/SCSS → 直接塞进项目
```

问题：

- 没有模块边界
- 没有字段边界
- 没有复用边界
- 不利于接入 schema
- 不利于接入 Supabase
- 后期维护成本极高

### 正确用法

```text
Figma / TRAE
→ 看设计
→ 拿素材
→ 你手动拆模块
→ 你手动定字段
→ 用 mock data 写组件
→ 校验字段
→ 固化 schema
→ defaults
→ 接 Supabase
```

## 6. 结合你当前项目，`.figma` 应该怎么用

你现在的正确工作流是：

```text
1. 看设计稿（结构理解）
2. 人工决定字段（最关键）
3. 用 mock data 写组件
4. 校验字段是否合理
5. 固化 schema
6. 后面再接 Supabase
```

这套流程和 `.figma/` 的最佳结合方式如下。

### Step 1：用 `.figma` 看结构，不写代码

重点看：

- 页面被分成哪几个大模块
- 模块内部有哪些稳定内容
- 哪些部分是“内容字段”
- 哪些部分只是“视觉表达”

例如首页可以先拆成：

- Hero
- Services
- Quality
- Certification
- Industry
- Why Choose Us
- Order Process
- FAQ

### Step 2：你人工决定字段

这是最关键的一步。

字段应该由你来控制，而不是由 AI 或导出代码来决定。

例如 Hero 模块，合理字段可能是：

- `title`
- `subtitle`
- `button_text`
- `button_link`
- `highlights`
- `background_image`

而不建议把这些做成字段：

- 每个字的颜色
- 每个 icon 的精确坐标
- 动画时长
- 所有视觉细节都参数化

原则：

- 字段只保留“真正影响内容管理”的部分
- 视觉表达尽量留在组件和样式里

### Step 3：用 mock data 写组件

例如：

```tsx
<Hero data={mock.hero} />
<Services data={mock.services} />
<Faq data={mock.faq} />
```

这一步的目的不是“做一整页静态 HTML”，而是：

- 验证字段是否够用
- 验证组件边界是否合理
- 验证哪些内容适合进入 schema

### Step 4：字段校验后再固化 schema

只有当 mock data 驱动组件已经跑顺后，才适合固化：

- `content/schemas/*.ts`
- `content/defaults/*.ts`

这样做的好处是：

- schema 更稳定
- 不容易出现写完 schema 又反复改字段名
- 后面接 Supabase 映射会更顺

### Step 5：最后再接 Supabase

当组件、字段、schema 都稳定后，再接：

- `pages`
- `seo_meta`
- `page_content`
- `global_sections`

这时数据库只是“把已经验证过的数据结构持久化”，而不是反过来逼着前端改结构。

## 7. 一句话判断：`.figma` 里的东西到底是不是“能用代码”

答案是：

**不是“不能用”，而是“不能直接当正式项目代码用”。**

更准确地说：

- 它是“设计稿导出物”
- 它是“参考输入”
- 它是“素材来源”
- 它不是“最终源码”

## 8. 推荐落地规则

建议把 `.figma/` 在项目中的定位固定为：

### 允许用途

- 看结构
- 看视觉
- 提取素材
- 校对细节

### 禁止用途

- 整页复制进正式项目
- 直接作为组件体系基础
- 直接作为 schema 设计依据

### 最佳实践

```text
.figma 只负责：
看设计 + 拿素材

正式项目负责：
模块拆分 + 字段决策 + mock data + schema + 数据接入
```

## 9. 最终结论

当前项目里，`.figma/` 目录应该被理解为：

**“设计稿导出缓存层”**

而不是：

**“前端源码层”**

对你这个项目来说，最正确的使用方式是：

```text
利用 `.figma/` 做结构理解和素材提取，
由你人工决定字段，
再用 mock data 组织组件，
最后固化 schema 并接入 Supabase。
```
