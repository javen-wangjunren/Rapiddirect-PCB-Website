# 总览：当前后台用什么方式做的

后台采用「Astro 路由 + React islands + Tailwind」的方式开发，不套用现成 Admin 模板/管理端框架，目标是先把编辑闭环跑通，并尽量还原 WordPress 后台的编辑体验。

闭环目标（事实）：

```text
后台编辑体验 → Supabase 保存 → 前台 /ems/ 刷新变化
```

核心策略：
- Astro 页面只负责路由与壳（layout），业务逻辑在 React 组件里完成
- 权限采用“前端 session guard + 数据库 RLS 兜底”
- 数据结构化：content_json（schema 驱动）+ defaults merge，支持渐进完善字段

# 技术栈与边界

## 前端栈

- Astro（路由与 Layout）
- React（后台交互与表单编辑器）
- Tailwind（UI 样式）

依赖层面没有引入现成 Admin UI 框架或组件库（例如 Radix/Headless UI/shadcn/React Admin 等），主要使用 React 内建 hooks + Tailwind class。

## 后端/数据栈

- Supabase
  - Auth：登录 session
  - DB：pages / page_content / seo_meta
  - 安全：RLS policies（写权限核心兜底）

## 为什么不直接用现成模板/框架

因为目标不是“快速出一个 dashboard”，而是“做一个 WP 心智模型的编辑器”：
- WP 的编辑体验重点在“页面身份（Title/Slug）+ 内容模块（Tabs/Accordion）+ 右侧 Publish/Settings/SEO”
- 常见 Admin 框架更擅长标准 CRUD（列表/详情/表单），对“schema 驱动的模块化编辑器”适配并不天然
- 模板能让 UI 立刻更像产品，但也会把布局与组件体系绑定，反而影响后续按 WP 方式迭代编辑器

# 路由与页面结构（Astro 壳 + React 岛）

## base 与路径

后台统一挂在 `/ems/*` 下，来自 Astro 的 `base: '/ems'` 配置。

## 典型后台页面模式

Astro page 通常只做 3 件事：
- `export const prerender = false;`
- 使用 AdminLayout 渲染 WP 风格壳
- 挂载 React 组件：`client:load`

示例页面（事实）：
- `/ems/login/`：登录入口
- `/ems/admin/pages/`：Pages 列表与创建入口
- `/ems/admin/pages/edit/?slug=...`：通用编辑入口
- `/ems/admin/pages/ems/`：EMS 首页快捷入口（等价于 edit /ems/）

# Session Guard（登录态校验）与安全模型

## 前端 Guard（体验层）

后台 React 页面在 `useEffect` 内调用 `supabase.auth.getSession()`：
- 无 session：跳转 `/login/`
- 有 session：继续加载/保存数据

这种方式贴近 WP 体验（“未登录直接回登录页”），实现成本低，能快速跑通闭环。

## 数据库 RLS（安全兜底层）

前端 guard 只解决体验，不解决安全；真正的写权限由 Supabase RLS 决定：
- 前台（anon key）：只读 published
- 后台（authenticated）：可读写 pages/page_content/seo_meta

结论：当前模型属于“客户端拦截 + 数据库强约束”。

# 数据模型与 CRUD 边界

## 三表复用（最小闭环）

不引入新 CMS 表，优先复用：
- pages：slug/title/template_type/status（页面骨架）
- page_content：content_json（结构化内容）
- seo_meta：meta/og/noindex 等（SEO 配置）

## 数据访问分层

建议保持“组件不直接写 SQL/链式 supabase 调用”的原则：
- React 页面/组件负责：状态、交互、组装 payload
- lib/supabase/*Queries 负责：查询与保存 bundle（读写三表）

# 编辑器的产品结构（对齐 WP 心智模型）

后台编辑器由 3 个“层级”组成：

1) 页面身份（不进入 Tabs）
- Title（大输入框）
- Slug / Permalink（可编辑）

2) 主内容区（内容模块）
- Tabs：模块分组（减少心智负担）
- Accordion：每个模块折叠（减少纵向滚动成本）
- SchemaForm：schema 驱动字段编辑（string/boolean/object/array）

3) 右侧栏（发布与页面级设置）
- Publish：draft/published、保存、发布/更新
- Page Settings：template_type、noindex 等
- SEO：独立卡片（页面级配置，不塞进内容模块）

# defaults + DB 覆盖（前台/后台协作方式）

前台渲染策略：
- DB 有值：优先用 DB
- DB 缺字段：用 defaults 补齐（页面不崩）

后台编辑策略：
- 对关键模板（例如 ems_home），编辑器加载时做 defaults merge，确保 schema 字段完整
- 对未完成模板，先用 JSON 占位编辑，保证闭环先跑通

价值：允许后台“渐进式完善字段”，而不是一次性做完整 CMS。

# 当前实现的工程化模式（事实总结）

- 主要状态管理：React hooks（useState/useEffect/useMemo），无全局 store
- 路由拼接：统一走 BASE_URL helper，避免手写 `/ems`
- UI：Tailwind + 自研组件（尚未系统化 design tokens 与 UI primitives）

# 后续优化方向（建议优先级）

## P0：不引入大框架，先把 UI 设计系统收口

目标：让后台从“纯开发感”变成“产品感”，同时不改变整体架构。

- 建立后台 tokens（建议直接对齐 WP 风格，偏浅色）
  - 色板：bg / text / border / brand link / danger / success
  - 间距与圆角：统一 input/card/button 的尺寸与密度
  - 字体与字号：标题/正文/辅助信息的层级规则
- 抽 UI primitives（内部组件）
  - Button / Input / Textarea / Select
  - Card / Table / Badge
  - Tabs / Accordion / Modal / Toast
- 统一交互细节
  - hover/focus/disabled/loading/danger
  - 键盘可达性（focus ring、Tab 顺序）

## P1：编辑体验更像 WP（提升可运营性）

- Dirty 状态与离开提示（未保存离开确认）
- 保存体验：保存中状态、保存失败可重试、最近保存时间提示
- 发布流更清晰：draft/published 的状态提示、发布确认
- 模块编辑效率：一键折叠/展开、复制模块、排序（后续可做拖拽）

## P2：SchemaForm 能力补齐（提升扩展速度）

- 字段级校验：required/min/max/pattern 与错误提示
- 字段说明 helpText/placeholder 的统一呈现
- 媒体选择器从 URL 走向 Media Library（有库后可选图、预览、替换）
- Array/Object 编辑体验：更清晰的增删改、排序、默认展开策略

## P3：工程化与复用（避免页面增多后重复劳动）

- 抽通用 guard：`useAdminSessionGuard()` 或 `AdminGate` 组件，减少每页重复代码
- 抽数据 hooks：`usePageBundle(slug)`、`useSaveBundle()`，统一 loading/error/toast
- 统一路由生成：持续使用 `getHref/getAssetPath`

## P4：是否要引入现成框架（建议）

不建议直接上“整套 Admin 模板/框架”来替换当前结构；更稳的路线是：
- 保持现有 WP 三栏心智模型与 schema 编辑器不变
- 只按需引入“交互原语库”（Tabs/Dialog/Popover/Toast 等）来降低交互维护成本
- 或继续自研，但需要配套设计系统与可访问性规范

# 快速自查清单（每次迭代前后）

- 路由是否全部通过 BASE_URL helper 生成（避免 hardcode /ems）
- 未登录是否都会被正确重定向到 /login
- 保存失败是否对用户可见且可重试
- 编辑器是否存在“DB 缺字段导致页面崩”的风险（defaults merge 是否覆盖）
- 表单控件是否具备清晰 focus 样式与键盘可达性
