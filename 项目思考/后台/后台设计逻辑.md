# 目标：先把“编辑闭环”跑通

我们优先完成的是一个可用的后台编辑闭环，而不是一个完整 CMS 系统：

```text
后台编辑体验 → Supabase 保存 → 前台 /ems/ 刷新变化
```

因此设计上遵循 3 个原则：

1. 先把编辑体验与信息架构做好（WordPress mental model），再逐步补功能。
2. 不引入拖拽式 builder；内容依然是 schema 驱动的结构化字段。
3. 页面级配置与内容模块分区，避免“所有字段堆到一起”。

---

# 信息架构：WordPress 风格的三栏心智模型

后台编辑器采用 WordPress 类似布局，但不追求像素级复刻，重点是“好用”：

## 1) 左侧 Sidebar（全局导航）

- 负责“我在哪/我要去哪里”
- 典型条目：Pages（列表）、Add New（创建）、View EMS（前台）
- 价值：随着页面增多，编辑入口不再依赖硬编码单页（例如只做 /admin/pages/ems）

## 2) 顶部（页面身份）

- Title（大输入框）
- Slug / Permalink（可编辑）

这两个字段属于“页面身份”，不进入 Tabs。

## 3) 主内容区（内容模块）

- 只对“内容模块”使用 Tabs + Accordion
- Tabs 的作用是分组；Accordion 的作用是减少纵向滚动成本

当前 EMS 首页（ems_home）已做到：
- Tab 分组（Overview/Capabilities/Trust/FAQ）
- 每个模块 Accordion 展开编辑

## 4) 右侧栏（发布与页面级设置）

- Publish：draft/published、保存草稿、发布/更新
- Page Settings：template_type、noindex 等
- SEO：单独卡片，不塞进内容模块 Tabs（因为它是页面级配置）

---

# 数据与边界：不新建 CMS 表，复用现有三表

我们不先引入新表 `cms_pages`，而是复用现有的最小闭环三表：

- `pages`：slug/title/template_type/status（页面骨架）
- `page_content`：content_json（结构化内容）
- `seo_meta`：meta/og/noindex（SEO 配置）

这有两个好处：

1. 后台编辑的输出直接影响前台渲染（同一数据源），闭环短。
2. 不需要先做一次数据迁移，降低试错成本。

---

# 前台与后台的配合方式：defaults + DB 覆盖

前台 `/ems/` 的渲染遵循：

- DB 有值：优先用 DB
- DB 缺字段：用 defaults 补齐（避免因为后台还没做完所有字段导致页面崩）

这使后台可以“渐进式完成表单”：

```text
先支持 hero.title 等关键字段 → 再扩展到全模块
```

---

# 路由设计：用 query 传 slug，避免动态路由处理“带 / 的 slug”

因为页面 slug 本身包含 `/`（例如 `/ems/`），如果用路径参数会很难做一个通用编辑路由。

所以采取：

- 登录：`/ems/login/`
- Pages 列表：`/ems/admin/pages/`
- 通用编辑：`/ems/admin/pages/edit/?slug=/ems/`
- EMS 首页快捷：`/ems/admin/pages/ems/`（等价于 edit /ems/）

---

# 权限与安全：后台用 authenticated，前台用 published

权限策略分两层：

1. 前台（anon key）：只读 published（已有策略）
2. 后台（登录用户 authenticated）：可读写 pages/page_content/seo_meta（已加 RLS policy 迁移）

后台编辑器在前端侧做 session guard：

- 未登录 → 跳转 /login
- 已登录 → 读取/保存

# 当前实现状态（事实）

- 已有登录页 `/ems/login/`
- 已有 admin shell（顶部栏 + 左侧 sidebar）
- 已有 Pages 列表 `/ems/admin/pages/`（可创建 draft page）
- 已有通用编辑 `/ems/admin/pages/edit/?slug=...`（读写三表）
- `ems_home` 已按 schema 友好编辑；`ems_service` 先用 JSON 占位
- 明确不做：前台可视化编辑、drag builder、pcb-assembly 优先推进

# 下一步要做什么（推荐顺序）

1. 让服务页跑通 “动态渲染闭环”：
   - 先做 `/ems/pcb-design/` 的 seed + 前台读取渲染
2. 把后台从“能编辑”推进到“可运营”：
   - Pages 列表加搜索/筛选（draft/published）
   - `ems_service` schema 表单化（替换 JSON 占位）
3. SEO 与上线必要项：
   - canonical/sitemap/robots
4. 全局内容（Phase 6/7）：
   - `global_sections` + 后台编辑入口
