# EMS 子站开发规范（执行版）

本文档是本项目的“不可违反规范”，用于约束：URL 体系、SEO 输出、资源路径、数据与安全边界、目录与编码约定、上线自检。

关联文档（用于理解背景与架构边界）：

- `项目介绍.md`
- `技术栈.md`
- `项目结构.md`

## 1. 项目目标与边界

目标：在主域路径下构建 EMS 子站，正式入口为：

```
https://rapiddirect.com/ems/
```

必须满足：

- SEO 继承主站权重（canonical/sitemap 等必须输出主域）
- URL 体系与 WordPress 规划一致（仅由反向代理接入，不改变对外路径）
- 可通过 Cloudflare 按 `/ems/*` 分流稳定运行
- 页面可模块化迭代，避免在 WordPress 内继续堆复杂页面

## 2. 基础配置（必须严格遵守）

### 2.1 Astro 配置（固定模板）

```
// astro.config.mjs
export default defineConfig({
  site: 'https://rapiddirect.com',
  base: '/ems',
  trailingSlash: 'always',
  build: { format: 'directory' }
})
```

### 2.2 强制规则

```
1) 所有页面路径必须以 /ems/ 开头
2) 所有页面必须带尾斜杠 /
3) 禁止出现无斜杠版本 URL
4) 禁止使用根路径资源（/xxx.css、/_astro/* 等）
```

## 3. URL 与路由规范（极其重要）

### 3.1 标准 URL 结构

```
/ems/
/ems/pcb-manufacturing/
/ems/pcb-assembly/
/ems/pcb-design/
/ems/components-sourcing/
```

### 3.2 允许/禁止

```
✅ 允许：
/ems/xxx/

❌ 禁止：
/ems/xxx
/xxx/
/ems//xxx
```

## 4. 内部链接规范

### 4.1 必须使用统一格式

```
<a href="/ems/pcb-assembly/">PCB Assembly</a>
```

### 4.2 禁止行为

```
❌ /pcb-assembly/
❌ https://临时域名/ems/xxx/
❌ 相对路径 ../xxx
```

## 5. 项目结构与文件约定

项目结构以 `项目结构.md` 为准。当前仓库文档位于 `项目思考/`，如果后续统一迁移到 `docs/`，规范内容不变，只是目录位置变化。

### 5.1 目录职责（必须理解）

- `src/pages/`：路由入口层，只做“拿数据 + 选择布局 + 组合本页模块”
- `src/layouts/`：全局布局层，集中处理 head/导航/footer/SEO 注入入口
- `src/components/`：组件层，按页面与通用组件划分
- `src/lib/`：非 UI 逻辑（Supabase 查询、SEO 工具、env、通用工具）
- `src/content/schemas/`：字段结构定义（内容 schema）
- `src/content/defaults/`：默认内容（fallback），用于本地开发和空数据兜底

### 5.2 命名规范

- 路由文件：`kebab-case.astro`（如 `pcb-assembly.astro`）
- 组件文件：`kebab-case.tsx`（如 `hero-banner.tsx`、`faq-item.tsx`）
- 导出组件：建议 `PascalCase`（文件名 kebab，导出名 Pascal，便于识别）
- 内部链接：始终使用 `/ems/.../` 的绝对路径字符串
- 页面文件名必须与对应页面 slug 的最后一级一致
  - `/ems/pcb-assembly/` → `pages/pcb-assembly.astro`
  - `/ems/components-sourcing/` → `pages/components-sourcing.astro`
  - 首页例外：`/ems/` → `pages/index.astro`

### 5.3 命名冲突规避（新增）

```
1) 页面专属组件必须放在对应页面目录下
   例如：
   components/pcb-assembly/hero.tsx
   components/pcb-design/hero.tsx

2) common 目录内的组件名称必须体现通用语义
   例如：
   section-heading.tsx
   stat-card.tsx
   faq-item.tsx

3) 禁止在 common 与页面目录中使用同名但不同职责的文件，除非语义完全一致

4) content schema 文件名必须与页面 slug 或页面名一致
   例如：
   content/schemas/pcb-assembly.ts
   content/defaults/pcb-assembly.ts

5) types 文件名必须按领域命名，不按临时业务名命名
   例如：
   page.ts / seo.ts / content.ts
   禁止：
   temp.ts / data.ts / utils2.ts
```

### 5.4 `common` 组件抽离条件

只有当一个组件在 2 个及以上页面复用，且结构与职责一致时，才允许抽入 `common`。

禁止：

- 因为“以后可能会复用”而提前抽象
- 把页面专属模块为了省事放进 `common`
- 让 `common` 变成无法判断职责的杂项目录

## 6. 页面与模块开发规范（Template-driven）

### 6.1 正式业务内容不得直接写死在组件中

原则：页面文案、列表项、图片等正式业务内容应来自 `content_json`（后续来自 Supabase）。

开发阶段允许：

- 使用 `content/defaults` 作为 fallback
- 使用局部 mock 数据进行占位
- 先完成静态页面以便对齐设计稿

上线前要求：

- 所有正式内容必须迁移到 `content_json` / Supabase
- 组件内不得残留仅供开发阶段使用的正式文案写死内容

### 6.2 `pages` 层只负责组合，不负责复杂渲染

`pages/*.astro` 应保持“薄”：

- 获取页面数据（后续：`pages + seo_meta + page_content`）
- 将数据传入组件
- 不在 page 文件里写复杂样式与大量 HTML

### 6.3 模块组件输入统一

模块组件统一接收：

- `data`：对应模块的内容分支（来自 `content_json`）

组件内部禁止：

- 直接读取 Supabase（查询集中放 `lib/supabase`）
- 自己拼 canonical/sitemap（SEO 集中放 `lib/seo` 或 layout 注入）

### 6.4 组件边界规则

页面级布局（section、背景、上下间距、整体栅格）优先由页面组件或页面专属模块控制；小组件只负责自身内容，不负责整页布局。

适用原则：

- 页面负责“结构与节奏”
- 模块负责“该 section 的完整渲染”
- 小组件负责“可复用的小单元”

避免：

- 小组件内部私自加大段页面级外边距/背景块
- 一个小组件同时承担页面结构 + 内容卡片 + 数据获取三种职责

### 6.5 配置与内容分离

必须区分：

- 全局配置：如站点信息、全局 CTA、认证墙、全局联系信息
- 页面内容：如某个页面的 hero、intro、faq、process

要求：

- 全局配置不得混入单页 `content_json`
- 单页内容不得反向承担全局配置职责
- 可复用内容优先抽为 `global_sections` 或单独配置项，不重复建多份

## 7. 开发流程原则（新增）

### 7.1 视觉优先

开发顺序必须优先遵守：

```text
设计稿 → 模块拆分 → 人工确定字段 → 组件开发（mock data）
→ 校验字段 → schema 定义 → defaults → Supabase 接入
```

也就是：

```text
Figma / 原型
→ 结构理解
→ 模块拆分
→ 人工确定字段
→ 用 mock data 写组件
→ 校验字段是否合理
→ schema 定义
→ defaults fallback
→ Supabase 接入
→ 验证
```

目的：

- 字段由人工控制，而不是由 Figma 导出物或 AI 自动推导
- 先把组件边界与数据接口想清楚，再接数据层，避免反复返工
- 避免把整个设计稿直接翻译成一整页不可维护代码

### 7.2 `.figma` / TRAE 使用边界

`.figma/` 目录中的内容本质上是“设计稿导出物”，不是项目正式源码。

正确用途：

- 看页面结构与视觉层级
- 拿图片、SVG、背景图、图标等素材
- 辅助判断模块边界、间距、排版、视觉样式

禁止用途：

- 直接把整页 `index.jsx + index.module.scss` 搬进项目作为正式实现
- 让 Figma 导出代码决定页面结构、字段结构或 schema
- 因为导出代码能跑就把它当成可维护组件体系

原则：

- TRAE 只用来“看设计 + 拿素材”
- 正式代码必须按模块手动组织
- 字段必须由人工决定，不由导出代码反向推导

### 7.3 单模块闭环

一次只完成一个页面模块的闭环，不并行铺太多模块。

推荐顺序：

```text
一个页面 / 一个模块
→ 模块拆分
→ 人工确定字段
→ mock data 组件开发
→ 字段校验
→ schema
→ content/defaults
→ 渲染接线
→ 验证
```

禁止：

- 同时并行铺多个模块但都处于半成品状态
- 页面全部切完才回头统一补 schema 或默认数据

## 8. 内容 schema 规范（强制）

`src/content/schemas/*` 用于定义每个页面或模块的数据结构，作为：

- Supabase `page_content.content_json` 的契约
- 组件 props 的类型来源
- 内容变更时的沟通依据

要求：

- schema 结构必须稳定、可扩展（新增字段不破坏旧字段）
- 列表类字段统一使用数组（如 `items[]`、`steps[]`）
- 图片字段统一为 URL 字符串（后续可扩展为对象，但需要版本迁移策略）
- schema 变更优先新增字段，不直接删除或重命名已有字段
- 如必须改名，必须同步更新：
  - `content/defaults`
  - 组件 props
  - Supabase 映射逻辑
  - 已有测试数据 / seed 数据

### 8.1 `content_json` 命名规范（新增）

```
1) content_json 顶层 key 使用模块语义名
   例如：
   hero / intro / process / faq / cta

2) 字段命名统一使用一种风格，项目内不得混用
   推荐：snake_case
   例如：
   primary_button_text
   primary_button_link

3) 列表字段统一使用复数语义名
   例如：
   items / steps / cards / faqs
   禁止：
   list1 / data / module_data

4) 布尔值字段统一使用 is_ / has_ / use_ 前缀
   例如：
   is_enabled
   has_badge
   use_global_cta
```

### 8.2 数据结构原则

- 优先语义化命名，避免临时命名
- 优先扁平结构，避免过深嵌套
- 字段尽量非必填，通过默认值与 fallback 保证兼容
- 结构升级时优先追加字段，避免随意改名破坏旧数据

## 9. 静态资源与图片规范（/ems 子路径兼容）

### 9.1 构建产物路径约束（必须满足）

```
/ems/_astro/*.css
/ems/_astro/*.js
```

禁止出现：

- `/_astro/*`（少了 `/ems` 前缀）
- 页面中引用临时域名资源

### 9.2 资源放置策略

- 优先使用 `src/assets/` 走构建资源管线，减少子路径部署 404 风险
- `public/` 只放极少量固定资源（如 favicon、robots.txt），并谨慎使用根路径引用

## 10. SEO 规范（核心）

### 10.1 Canonical（逐页生成）

示例：

```
<link rel="canonical" href="https://rapiddirect.com/ems/pcb-assembly/" />
```

强制规则：

```
1) 每个页面 canonical 必须指向自己（主域 + 正式路径）
2) 禁止所有页面 canonical 指向 /ems/
3) 必须使用 https://rapiddirect.com（不带临时域名、避免 http/www 混用）
```

### 10.2 Sitemap

```
只允许出现：
https://rapiddirect.com/ems/xxx/
```

禁止出现：

- 临时域名
- 不带 `/ems/` 的路径

### 10.3 Robots（临时域名）

临时域名/测试域名必须禁止抓取：

```
User-agent: *
Disallow: /
```

### 10.4 页面语义结构

每个页面至少包含：

- title
- meta description
- 仅 1 个 H1
- 清晰的 H2/H3 层级

禁止：

- 多个 H1
- 空 title/description
- 用纯 div 堆页面导致语义缺失

## 11. 缓存策略规范（Cloudflare）

按类型区分：

```
HTML 页面：短缓存或绕过缓存
/ems/_astro/*（hash 静态资源）：长缓存（例如 1 年）
API / 表单：禁止缓存
```

## 12. Supabase 规范（安全优先）

### 12.1 前端调用规则

```
1) 允许浏览器直接调用 Supabase
2) 必须使用 public anon key
3) 禁止暴露 service role key
```

### 12.2 RLS 与 CORS（强制）

- 所有表必须开启 RLS，并按读写场景设计 policy
- CORS 必须允许主域：`https://rapiddirect.com`

## 13. 禁止事项（任何阶段都不能做）

```
❌ 页面/资源/OG/结构化数据中出现临时域名
❌ 使用 / 而不是 /ems/ 作为路径
❌ 忽略尾斜杠
❌ 所有页面 canonical 相同
❌ 静态资源路径不带 /ems/
❌ 混用 http/https 或 www/非 www
❌ 在组件里直接写 Supabase 查询
❌ 在组件里写死正式业务文案（应进 content_json 或 defaults）
```

## 14. 上线前自检 Checklist（必须逐项通过）

```
[URL]
✔ 是否全部 /ems/.../
✔ 是否全部带尾斜杠

[资源]
✔ CSS/JS 是否全部 /ems/_astro/
✔ 图片与 OG 资源是否不存在临时域名

[SEO]
✔ 每页 title / description 完整
✔ 每页 canonical 指向自身主域正式 URL
✔ sitemap 仅输出主域 /ems/ 路径

[链接]
✔ 内链是否全部使用 /ems/.../
✔ 是否无相对路径 ../xxx

[缓存]
✔ HTML 未强缓存
✔ /ems/_astro/* 已长缓存
✔ API/表单不缓存

[数据与安全]
✔ 前端仅使用 anon key
✔ 不存在 service role key 泄露
✔ RLS 已开启并配置 policy
✔ CORS 允许 https://rapiddirect.com
```

## 15. 一句话执行总结

```
你正在开发一个部署在 /ems/ 子路径下的 Astro 网站。
必须：路径带 /ems/、URL 带尾斜杠、canonical 指向主域正式 URL、资源路径带 /ems/、页面中不出现临时域名。
```
