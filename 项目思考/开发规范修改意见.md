# 现在最值得补的内容

## 一、补“命名冲突规避规则”

建议你在规范里新增一节：

## 5.3 命名冲突规避（建议新增）

```text
1) 页面专属组件必须放在对应页面目录下
   例如：
   components/pcb-assembly/hero.tsx
   components/pcb-design/hero.tsx

2) common 目录内的组件名称必须体现通用语义
   例如：
   section-heading.tsx
   stat-card.tsx
   faq-item.tsx

3) 禁止在 common 与页面目录中使用同名但不同职责的文件，除非语义完全一致

4) content schema 文件名必须与页面 slug 或页面名一致
   例如：
   content/schemas/pcb-assembly.ts
   content/defaults/pcb-assembly.ts

5) types 文件名必须按领域命名，不按临时业务名命名
   例如：
   page.ts / seo.ts / content.ts
   禁止：
   temp.ts / data.ts / utils2.ts
```

## 二、补“字段 key / content_json 命名规范”

你现在规范里说了 schema 要稳定，但没明确字段命名方式。

建议补：

```text
1) content_json 顶层 key 使用模块语义名
   例如：
   hero / intro / process / faq / cta

2) 字段名统一使用 snake_case 或 camelCase，只能选一种，项目内保持一致
   推荐：snake_case
   例如：
   primary_button_text
   primary_button_link

3) 列表字段统一使用 items / steps / cards / faqs 等复数名
   禁止：
   list1 / data / module_data

4) 布尔值字段统一使用 is_ / has_ / use_ 前缀
   例如：
   is_enabled
   use_global_cta
```

## 三、补“组件边界规则”

这个其实你的旧规范里有很强的参考价值。

你之前前端原则里最值得借鉴的是：

### **组件纯度优先**

组件不要带页面级结构，外层负责 section、间距、标题区。

这个对你现在的 Astro 项目也非常适用，可以简化成一句：

```text
页面级布局（section、背景、上下间距）优先由页面组件控制；
小组件只负责自身内容，不负责整页布局。
```

不用照搬得太重，但这个思想值得留下。

# 你那三个旧文件，哪些值得借鉴？

---

## 1. `core-operational-principles.md`

这个文件最值得借鉴。

尤其这两条：


这条非常适合你现在。

**建议直接借鉴进新规范。**

## 2. `frontend-dev-principles.md`

## 3. `backend-dev-principles.md`

这个文件里最值得借鉴的是“思想”，不是“实现”。

### 值得借鉴的：

* DRY：全局内容不要重复建
* 配置与内容分离
* 命名语义化
* 字段非必填
* 数据结构尽量扁平

这些思路都适用于 Supabase。 

### 不建议直接迁移的：

* ACF Clone
* Tab → Accordion → Field
* Classic Editor / Gutenberg 管理
* Image Return ID 这类 WP 特定约束

这些是 WordPress 体系内的做法，不要硬搬。
