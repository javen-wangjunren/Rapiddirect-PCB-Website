# EMS 子站开发路线图（执行版）

本文档定义本项目从 0 到上线的开发路线图，目标是把需求拆成可执行阶段、明确交付物、设定验收标准，并与 `项目介绍.md`、`技术栈.md`、`项目结构.md`、`开发规范.md` 保持一致。

## 1. 项目目标

构建 EMS 子站，正式访问路径为：

```
https://rapiddirect.com/ems/
```

项目目标：

- 使用主域 `/ems/*` 子路径承接 SEO
- 采用 Astro + React + Tailwind 构建前端
- 使用 Supabase 承载内容、SEO、媒体与表单数据
- 采用 Template-driven 模式，不做自由拖拽式 page builder
- 先完成首个页面 `/ems/` 的设计稿落地，再扩展到其它服务页

## 2. 当前范围（MVP）

首期页面范围：

```
/ems/
/ems/pcb-manufacturing/
/ems/pcb-assembly/
/ems/pcb-design/
/ems/components-sourcing/
```

当前最优先页面：

- `/ems/` 首页

已知首页模块（当前已明确 8 个）：

1. Hero Banner
2. EMS Service
3. Quality Control
4. Certification
5. Industry
6. Why Choose Us
7. Order Process
8. FAQ

说明：

- 你提到首页共有 10 个模块，当前已明确 8 个；路线图执行时以“先完成已确认模块闭环，后补齐剩余模块”为原则

## 3. 总体技术架构

```
用户访问 rapiddirect.com/ems/*
        ↓
Cloudflare（按 /ems/* 分流）
        ↓
Cloudways 新 Application
        ↓
Astro + React + Tailwind 应用
        ↓
Supabase（内容 / SEO / 媒体 / 表单）
```

## 4. 路线图原则

### 4.1 执行原则

- 先拆模块，再由人工决定字段，再让组件吃 mock data
- 一次只完成一个页面或一个模块闭环
- 先首页，后扩展服务页
- 先前台可见结果，后后台编辑能力
- 所有开发必须遵守 `/ems/` 子路径、尾斜杠、canonical 与资源路径规范
- `.figma` / TRAE 只用于结构理解与素材提取，不直接作为正式页面代码

### 4.2 交付顺序原则

```
设计稿 → 模块拆分 → 人工确定字段 → 组件开发（mock data）
→ 校验字段 → schema/defaults → Supabase 接入
→ 动态渲染 → 后台编辑 → SEO/部署上线
```

## 5. 数据模型与模板方向（路线图基线）

### 5.1 Supabase 基础表

本项目路线图统一以以下表为准：

- `pages`
- `seo_meta`
- `page_content`
- `global_sections`

说明：

- 页面内容统一走 `page_content.content_json`
- 不采用旧稿中的 `page_sections` 作为主模型，避免与当前方案冲突

### 5.2 template_type 规划

首期统一使用：

```
ems_home
ems_service
```

含义：

- `/ems/` 使用 `ems_home`
- 其余服务页优先归入 `ems_service`，通过页面内容差异区分

## 6. 阶段规划（推荐执行顺序）

### Phase 0：项目初始化与基线确认

目标：把项目跑起来，并锁定“不会再反复变”的基础规则。

主要任务：

- 初始化 Astro + React + Tailwind 项目
- 配置 `site` / `base` / `trailingSlash` / `build.format`
- 建立项目目录结构（按 `项目结构.md`）
- 建立基础 layout、styles、assets、types、lib 目录
- 建立开发文档目录与 README

交付物：

- 可运行的 `apps/ems-site/`
- 基础目录结构与空页面路由
- 可复用的 `EmsLayout`

验收标准：

- 本地开发环境可启动
- `/ems/` 子路径配置正确
- 资源路径与页面路由不偏离开发规范

### Phase 1：首页模块拆分与字段决策

目标：围绕 `/ems/` 首页完成模块清单、人工字段决策与组件开发前置准备。

主要任务：

- 按设计稿拆分首页模块
- 人工决定每个模块真正需要的字段
- 剔除不应进入 schema 的视觉噪音字段
- 确认首页模块命名、组件命名、字段命名
- 形成首页 mock data 结构草案

交付物：

- 首页模块清单
- 首页字段清单
- 首页 mock data 草案

验收标准：

- 每个模块均有明确字段定义
- 字段由人工确认，不由 AI 或导出代码自动决定
- mock data 结构能直接用于组件开发

### Phase 2：首页组件开发（mock data）

目标：按模块开发首页组件，用 mock data 驱动，而不是先拼一整页静态 HTML。

主要任务：

- 根据设计稿逐模块实现首页组件
- 使用 mock data 驱动组件渲染
- 完成响应式布局
- 完成基础交互占位（如 FAQ 折叠可先静态或轻交互）
- 在组件开发过程中反向校验字段是否合理

交付物：

- 首页专属组件目录
- 首页 mock data 驱动页面
- 首页基础样式体系

验收标准：

- 首页视觉与设计稿高度一致
- 桌面端与移动端布局正常
- 组件边界清晰，不在页面中堆大量重复 HTML
- 页面不是由整页导出代码直接拼装而成

### Phase 3：首页 schema 固化与服务页骨架

目标：把首页已经验证过的字段结构固化成 schema/defaults，并开始搭其它 4 个服务页的骨架。

主要任务：

- 将首页 mock data 固化为 `content/schemas/ems.ts`
- 创建 `content/defaults/ems.ts`
- 提炼 `components/common/` 中真正复用的组件
- 建立 4 个服务页路由文件
- 为每个服务页建立页面专属组件目录
- 为各服务页先建立字段草案与 mock data 草案

交付物：

- 首页 schema
- 首页 defaults
- `components/common/`
- `components/pcb-assembly/`
- `components/pcb-design/`
- `components/pcb-manufacturing/`
- `components/components-sourcing/`
- 4 个服务页静态骨架

验收标准：

- 首页字段结构已稳定并固化
- `common` 组件只包含真实复用组件
- 服务页命名与 slug 一致
- 服务页结构可继续扩展，不与首页结构混乱耦合

### Phase 4：Supabase 基础接入

目标：完成数据库、存储与前端数据读取基础能力。

主要任务：

- 创建 `pages`、`seo_meta`、`page_content`、`global_sections`
- 配置 Storage bucket：`pcb-assets`
- 写 `lib/supabase/client.ts`
- 写 `lib/supabase/queries.ts`
- 写 `lib/supabase/mappers.ts`
- 初始化首页与一个服务页的测试数据

交付物：

- Supabase 表结构
- 前端查询封装
- 初始 seed 数据

验收标准：

- 可以通过 slug 查询页面
- 可以读取 `content_json`
- 可以读取对应 SEO 数据

### Phase 5：动态渲染闭环

目标：先让 `/ems/` 首页完成“数据库驱动闭环”，再扩展其它页面。

主要任务：

- 页面根据 slug 查询 `pages`
- 读取 `page_content.content_json`
- 读取 `seo_meta`
- 根据数据渲染首页模块
- 支持 `use_global_*` 或全局内容读取

推荐顺序：

```
先 /ems/
再 /ems/pcb-assembly/
再复制到其它服务页
```

交付物：

- 数据驱动首页
- 数据驱动至少 1 个服务页

验收标准：

- 修改 Supabase 数据可直接反映到页面
- 页面无硬编码正式业务内容残留
- SEO 信息可随数据变化而变化

### Phase 6：SEO 与全局内容能力

目标：把 SEO 与全局内容能力补齐，避免后期返工。

主要任务：

- 封装 canonical 逻辑
- 封装 meta title / description / OG 输出
- 生成 sitemap
- 配置临时域名 robots
- 建立 `global_sections` 的前端读取方式
- 抽离全局 CTA / 认证模块等可复用内容

交付物：

- `lib/seo/*`
- `robots.txt`
- sitemap 生成能力
- global sections 读取逻辑

验收标准：

- 每页 canonical 指向自身主域正式 URL
- sitemap 只输出主域 `/ems/` 路径
- 临时域名禁止抓取

### Phase 7：后台管理（简版 CMS）

目标：在不改代码的情况下可以更新页面内容与 SEO。

主要任务：

- 页面管理：title / slug / template_type / status
- SEO 管理：meta_title / meta_description / canonical_url / og
- 页面内容编辑：按 schema 编辑 `content_json`
- 全局模块管理：global CTA / certs / reviews 等

建议范围：

- 先做简版管理界面
- 不追求可视化拖拽
- 先满足“可编辑、可保存、可预览数据变化”

交付物：

- 简版内容管理后台
- 页面与 SEO 编辑能力

验收标准：

- 非开发人员可修改页面内容
- SEO 可独立编辑
- global sections 可单独维护

### Phase 8：表单、媒体与交互增强

目标：补全真实业务需要的询盘能力与基础交互。

主要任务：

- 表单提交流程接入 Supabase
- 文件上传接入 Storage
- FAQ、Tabs、步骤组件等局部 React 交互
- 必要的轻量动效（hover、计数、展开收起）

交付物：

- 可用的询盘表单
- 文件上传能力
- 基础交互组件

验收标准：

- 表单可成功提交并追踪
- 上传流程稳定
- React 仅用于局部交互，不接管 SEO 主内容

### Phase 9：部署、缓存与上线

目标：完成主域接入、缓存策略配置与上线前检查。

主要任务：

- Cloudflare `/ems/*` 路径分流
- Cloudways 应用部署
- 缓存规则配置
- 上线前 SEO 检查
- 主域路径访问验证

交付物：

- 可访问的预发 / 正式环境
- 路径分流规则
- 缓存配置

验收标准：

- `rapiddirect.com/ems/*` 正常访问
- `/ems/_astro/*` 长缓存生效
- HTML 与 API 不被错误缓存

## 7. 推荐开发节奏

### 7.1 最小可行顺序

如果按最稳的顺序推进，建议：

```
1. Phase 0
2. Phase 1
3. Phase 2
4. Phase 3（先固化首页 schema/defaults，再补服务页骨架）
5. Phase 4（只接首页）
6. Phase 5（首页闭环）
7. Phase 6
8. Phase 7
9. Phase 8
10. Phase 9
```

原因：

- 先把首页跑通，能最快验证“设计稿 → 模块 → 字段 → mock → schema → 数据 → 渲染”的整条链路
- 首页闭环后，服务页会复制得更快、更稳

## 8. 各阶段输出清单

### MVP 必须完成

- `/ems/` 首页设计稿落地
- 至少 1 个服务页完成数据驱动闭环
- Supabase 基础表与查询封装
- 每页独立 SEO
- 主域 `/ems/*` 路径部署可用

### 第二阶段增强

- 其余 4 个页面全部接入
- global sections 完整落地
- 简版 CMS
- 表单与上传

### 第三阶段进阶

- draft / publish
- preview
- 内容版本管理
- 更完整的后台体验

## 9. 路线图中的禁止事项

路线图执行过程中，禁止：

- 同时铺太多页面但没有任何一个闭环
- 先做后台，再回头补前台结构
- 在没有稳定 schema 前就大量写数据库数据
- 把 `page_content` 和旧的 `page_sections` 模型混用
- 为“未来可能复用”而过早抽象组件
- 直接把 `.figma` 生成的整页 `index.jsx + css/scss` 当正式实现

## 10. 一句话执行总结

```
本项目的正确推进方式是：
先从设计稿中拆模块，再由人工确定字段，用 mock data 写组件并校验字段，之后再固化 schema/defaults、接入 Supabase，最后复制这套模式到服务页、CMS 与上线部署。
```
