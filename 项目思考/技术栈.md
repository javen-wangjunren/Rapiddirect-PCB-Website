# EMS 子站技术栈与选型

## 0. 关键结论（先按这套做）

- **访问入口**：`rapiddirect.com/ems/*`（承接主站 SEO 权重）
- **流量接入**：Cloudflare 按路径 `/ems/*` 分流到 Cloudways 新应用
- **前端**：Astro（页面与 SEO 输出）+ React（局部交互）+ Tailwind（样式体系）
- **数据层**：Supabase（Postgres + Storage），用于内容/SEO/表单/媒体
- **渲染策略（默认）**：以静态输出为主（SSG），只在需要时新增少量交互/接口

## 1. 项目定位与页面范围

本项目是 `rapiddirect.com` 主站下的 **EMS 业务子站**，规划页面：

```
https://rapiddirect.com/ems/
https://rapiddirect.com/ems/pcb-manufacturing/
https://rapiddirect.com/ems/pcb-assembly/
https://rapiddirect.com/ems/pcb-design/
https://rapiddirect.com/ems/components-sourcing/
```

约束（必须遵守）：

- 所有页面路径必须以 `/ems/` 开头
- 所有页面必须带尾斜杠（`/ems/xxx/`）
- 页面与资源中禁止出现临时域名（包括图片、OG、结构化数据、链接）

## 2. 选型原则（为什么是这套）

- **SEO-first**：title/description/canonical/sitemap 可靠输出，URL 体系稳定且一致
- **性能优先**：首屏 HTML 直出，静态资源长缓存，避免 WordPress 性能瓶颈
- **可维护**：页面按模板拆模块，内容走数据层，不把文案写死在组件里
- **接入友好**：Cloudflare 路径分流 + 反向代理让用户无感知、主域不变
- **安全边界清晰**：前端只用 Supabase anon key；敏感能力留在服务端/平台规则中

## 3. 总体架构（数据与请求流）

```
用户访问 rapiddirect.com/ems/*
      │
      ▼
Cloudflare（按 /ems/* 分流；缓存规则区分 HTML/静态资源/API）
      │
      ▼
Cloudways 新 Application（Astro + React + Tailwind）
      │
      ├─ 页面 HTML（SEO 主要内容直出）
      ├─ 静态资源（/ems/_astro/*，hash 资源长缓存）
      └─ 数据请求（Supabase：内容/SEO/表单/媒体）
```

## 4. 前端技术栈（明确职责）

### 4.1 Astro（核心）

- **用途**：路由、页面渲染、静态构建、SEO 输出（canonical/sitemap/OG 等）
- **默认策略**：页面主体内容静态输出；React 只做局部 islands
- **必须配置**：

```js
export default defineConfig({
  site: 'https://rapiddirect.com',
  base: '/ems',
  trailingSlash: 'always',
  build: { format: 'directory' }
})
```

### 4.2 React（只做交互，不做整站 SPA）

适用场景：

- FAQ 手风琴/折叠
- 表单（验证、上传、分步）
- 轻量交互（Tabs、筛选、动态计数）

禁止场景：

- 让 React 接管整页主体内容
- 用 React “渲染 SEO 主内容”（会增加爬虫/首屏风险与复杂度）

### 4.3 Tailwind（统一样式体系）

- 用于快速对齐设计稿 + 形成可复用的基础样式体系
- 建议沉淀基础组件层：Button / Container / Section / Card / Heading / FormField / FAQItem
- 设计稿导出的零散 CSS/SCSS 仅作为参考，最终以 Tailwind 体系收敛为主

### 4.4 TypeScript（推荐，默认开启）

- 目的：减少“模块字段 schema”变更带来的运行时错误
- 约束：组件 props 必须有类型，尤其是从 Supabase 读取的 `content_json`

## 5. 数据层选型：Supabase（内容 + SEO + 表单 + 媒体）

### 5.1 为什么用 Supabase

- Postgres 结构化存储适合页面/SEO/表单
- Storage 适合管理图片与附件（后续表单上传）
- RLS + anon key 支持前端安全访问（在规则控制下）

### 5.2 数据模型建议（与路线图一致）

- `pages`：slug、title、template\_type、status
- `page_content`：`content_json`（每页一份页面内容）
- `seo_meta`：meta\_title/meta\_description/canonical\_url/OG/noindex
- `global_sections`：全局复用模块内容（如全局 CTA、认证墙等）

### 5.3 安全规范（强制）

- 浏览器端仅使用 **anon key**，禁止使用 service role key
- 所有表必须开启 RLS，并按读写场景设计 policy
- CORS 必须允许主域：`https://rapiddirect.com`

## 6. SEO 技术规范（必须落地）

- **Canonical**：每页独立，指向主域正式 URL（例如 `https://rapiddirect.com/ems/pcb-assembly/`）
- **Sitemap**：只能输出主域 + `/ems/` 路径的 URL
- **Robots（临时域名）**：全站禁止抓取
- **语义结构**：每页 1 个 H1，H2/H3 层级清晰，SEO 主内容直接输出在 HTML

## 7. 静态资源与图片策略

### 7.1 资源路径规范（强制）

构建后资源必须能在子路径下访问：

```
/ems/_astro/*.css
/ems/_astro/*.js
```

禁止出现：

- `/_astro/*`（少了 `/ems` 前缀）
- 页面中引用临时域名资源

### 7.2 图片来源建议

- **固定资产**（logo、图标、通用装饰图）：随代码仓库管理
- **可运营资产**（可替换 banner、模块配图、OG image 等）：放 Supabase Storage，URL 由数据层配置

## 8. 部署与接入（Cloudways + Cloudflare）

### 8.1 为什么不优先用 Vercel

- 本项目的核心是“主域路径 /ems 分流 + 反向代理接入”，Cloudways 与现有主站运维链路更一致
- 便于与现有 Cloudflare/Cloudways 配置统一管理（缓存、规则、证书、日志）

### 8.2 Cloudflare 缓存策略（按类型分层）

- HTML 页面：短缓存或绕过缓存（保证内容更新可见）
- `/ems/_astro/*` hash 静态资源：长缓存（例如 1 年）
- API / 表单请求：禁止缓存

## 9. 工程化与质量（建议默认开启）

### 9.1 代码规范

- 统一格式化与 lint（避免多人/AI 修改风格漂移）
- 组件与模块不得写死业务文案（文案进 `content_json`）

### 9.2 运行环境（约束建议）

- Node LTS（建议 18+ 或 20+）
- 统一包管理器（推荐 pnpm；如团队已有 npm/yarn 则保持一致）

## 10. AI 执行约束（可直接当 Checklist）

```
1) 所有页面都在 /ems/* 且带尾斜杠
2) astro.config 必须配置 site/base/trailingSlash/build.format
3) canonical/sitemap/导航链接都输出主域正式 URL
4) 资源路径必须带 /ems（尤其是 /ems/_astro/*）
5) React 只做局部交互，不接管 SEO 主体内容
6) Supabase 只用 anon key；RLS 与 CORS 必须配置
7) Cloudflare 缓存按 HTML/静态资源/API 分层
```
