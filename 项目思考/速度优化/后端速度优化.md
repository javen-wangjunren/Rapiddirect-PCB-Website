# 后台（Pages / Media）加载 2–3s：速度优化分析（后端向）

## 现象

- 每次点击后台 `Pages` 或 `Media` 页面，都会出现约 2–3 秒的加载等待（页面骨架屏/空白期）。
- 该等待可能来自两类耗时叠加：
  - **路由导航/TTFB**：点击菜单会触发整页导航（非 SPA），如果当前部署形态是“按请求启动/Serverless”，会出现明显的 **冷启动** 或 **SSR 开销**。
  - **数据请求**：页面挂载后会调用 Supabase（PostgREST / Storage）拉取列表数据，网络往返 + DB 扫描 + payload 偏大也会带来 1–3 秒延迟。

## 补充：Media 页面截图里的 504（关键结论）

你截图中报错的资源形态是：

- `astro-*.js`
- `settings-*.js`
- `toolbar-*.js`
- `ui-library-*.js`
- Initiator 显示 `entrypoint.js`

这类 504 **不是 Supabase 查询慢**，而是 **Astro Dev Toolbar（开发工具栏）在 dev 环境注入的脚本资源加载失败**：

- dev 环境会注入：`/@id/astro/runtime/client/dev-toolbar/entrypoint.js`
- 然后它会继续拉一串 Vite 依赖 chunk（上面那些 `astro-* / toolbar-* / settings-* / ui-library-*`）
- 如果你的 dev 访问链路中间有反代/网关/超时/静态资源映射异常，就会出现这些脚本 504，从而拖慢页面加载与交互

对应处理（已落地在代码里）：

- 在 Astro 配置中关闭 Dev Toolbar：`devToolbar: { enabled: false }`
  - 文件：[astro.config.mjs](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/astro.config.mjs)
- 用 Network 判别：
  - `media_folders / media_assets` 返回 200 且耗时约 0.3–1.5s（你截图里就是这种），说明 DB/接口不是 504 主因
  - 504 集中在 `Type=script` 的 toolbar 相关资源，说明是 dev 注入链路问题

注意：

- `pnpm build && pnpm preview` 形态下不会注入 dev toolbar，通常不会出现上述脚本 504；因此如果线上也出现 504，需要单独抓取线上 Network 的 504 URL 来定位（更可能是反代超时/静态资源路径/base 配置）。

### 2) Pages 列表一次性拉取全量（未分页）

- UI 侧本地分页（pageSize=20），但数据侧一次性获取全部：

- 网络传输变慢（返回数组越大越慢）
- DB 端排序开销增大（尤其缺少 `updated_at` 索引时）
- 页面首次渲染等待变长（JS 解析 + React 渲染）

## 先把“慢在什么地方”拆清楚（必做）

目标：分辨 2–3 秒主要来自 “页面导航/SSR” 还是 “Supabase 数据请求/DB”。

1) 用 Chrome DevTools → Network
- 点击 `Pages` 菜单：
  - 看文档请求（`/ems/admin/pages/`）的 **TTFB** 是否接近 2–3 秒
  - 同时看随后发起的 Supabase 请求（`/rest/v1/pages?...`）耗时
- 点击 `Media` 菜单：
  - 同理看 `/ems/admin/media/` 的 TTFB
  - 看 `/rest/v1/media_folders`、`/rest/v1/media_assets` 的耗时（是否串行）

2) 两个快速判别结论
- 如果 **TTFB 本身就接近 2–3 秒**：优先处理“SSR/部署冷启动/静态化 admin”
- 如果 TTFB 很快，但 **Supabase 请求耗时 2–3 秒**：优先处理“查询/索引/分页/聚合”

## 优化方案（按优先级）

### A. 先消除“每次点击整页 SSR”的成本（结构性收益最大）

**A1. 能静态化的 admin 页面尽量 prerender**

`Pages` 列表与 `Media` 列表页面本身不依赖服务端数据（都在客户端拉取），理论上可改为 prerender：
- `apps/ems-site/src/pages/admin/pages/index.astro`：`prerender = true`
- `apps/ems-site/src/pages/admin/media/index.astro`：`prerender = true`
- `apps/ems-site/src/pages/admin/pages/ems.astro` 也可 prerender（slug 固定）

收益：
- 大幅降低点击菜单后的 TTFB（避免 SSR/冷启动）
- 让“感知速度”更多取决于数据请求本身，问题更可控

风险/注意：
- `edit.astro` 当前读取 `Astro.url.searchParams`（见 [edit.astro](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/pages/admin/pages/edit.astro#L1-L12)），静态化会失效；可改为在 React 组件中读取 `location.search`，使该页也能 prerender（可选）。

**A2. 进一步：将 admin 做成 SPA（单页壳 + 客户端路由）**

当前导航是 `<a>` 触发整页刷新（布局、CSS、JS 重载）。把 `/admin/*` 变成一个 React SPA（或 Astro + Client Router），可带来：
- 页面切换几乎瞬时（只换内容区）
- 列表数据可做内存缓存，返回时无需重新拉取

这是“体验层面最有效”的改法，但改动面更大，建议在 A1/A3 后再做。

### B. 把 Supabase 列表查询改成“分页 + 条件下推”（避免全量拉）

**B1. Pages 列表：后端分页 + 状态/搜索下推**

现状：全量拉取 + 前端过滤/分页（[AdminPagesIndex.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/AdminPagesIndex.tsx#L51-L115)）。

建议：
- 查询只取当前页（`range(from, to)`），避免返回全部 rows
- statusFilter 用 `.eq('status', ...)` 下推
- 搜索用 `ilike` 下推（并配合 pg_trgm 索引）

收益：
- 数据量稳定（每次 20 条），即使 pages 很多也不会变慢
- 服务器/DB 成本可控

**B2. Media 列表：分页 + 列选择 + 优化 unfiled/prefix 查询**

现状：`select('*') limit 200`（[adminMedia.ts](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/lib/supabase/adminMedia.ts#L76-L90)）。

建议：
- 改为显式列：`select('id,path,public_url,title,alt,mime_type,size,created_at')`
- 改为分页：`range(from, to)`，UI 的 pageSize=24 对齐
- folderFilter 的 prefix 查询依赖索引（见下文索引方案）
- `unfiled` 当前是 `not like '%/%'`，建议改为更可索引的字段（例如 `folder_id is null`）

### C. 给数据库补齐必要索引（通常是“后端性能”的主战场）

以下是“通用、收益明确”的索引（建议以新 migration 落地）。

**C1. pages 列表排序 + 状态过滤**

- `pages(updated_at desc)`：支持 `.order('updated_at', { ascending: false })`
- `pages(status, updated_at desc)`：支持 status 筛选 + 最新排序
- （可选）搜索：对 `slug/title` 做 `pg_trgm`（如果要做模糊搜索）

**C2. media_assets 列表排序 + 前缀过滤**

- `media_assets(created_at desc)`：支持最新排序
- `media_assets(path text_pattern_ops)`：让 `path like 'prefix/%'` 能走前缀索引
- （可选）如果会做 contains 搜索（path/title/alt）：`pg_trgm` 索引

**C3. 结构优化：给 media_assets 增加 folder_id**

当前 folder 是“靠 path 前缀表达”的，导致：
- `unfiled` 需要 `not like '%/%'`（难以索引）
- folder 查询依赖前缀 like

建议：
- `media_assets` 增加 `folder_id uuid null references media_folders(id)`
- 上传时由 UI 直接写入 `folder_id`
- 过滤变成：
  - 全部：无条件
  - unfiled：`folder_id is null`
  - folder：`folder_id = ?`

这能把“字符串模式匹配”变成“等值查询”，通常会明显提速。

### D. 减少往返次数（后端/网络层）

**D1. Media refresh 并行化**

目前 refresh 是串行的（folders → assets），可改为：
- 首次加载/`all`/`unfiled`：`Promise.all([folders, assets])`
- folder id 过滤：先用缓存的 folders 找 path，再拉 assets（或把 folder_id 方案落地后不需要依赖 folders）

**D2. 建立聚合接口（Edge Function / Worker）**

当需要“一个页面多表数据”（比如 Page Editor 需要 pages + page_content + seo_meta）：
- 可以用一个 Edge Function 把三次请求聚合成一次，顺便做缓存与鉴权

注意：这需要严格鉴权（验证 Supabase JWT / 或自有 session），避免暴露服务端密钥。

### E. 部署/区域：把服务端与 Supabase 放近一点

如果 Network 显示：
- Supabase 请求 RTT 很高（地理距离）
- 或 node SSR 端在不同区域导致跨区访问

建议：
- 让 SSR 部署区域与 Supabase Project 区域一致
- 或将 admin 静态化（A1）尽量绕开 SSR 区域问题

## 落地清单（建议按顺序）

1) 先测：区分 TTFB vs Supabase
- 记录两次点击（Pages、Media）的：文档请求 TTFB、Supabase 请求耗时、请求数量

2) 快速赢（不改表结构）
- Pages：后端分页 + statusFilter 下推
- Media：并行拉 folders/assets、显式列、分页
- DB：补 `pages(updated_at)`、`media_assets(created_at)` 索引

3) 中期优化（改表结构）
- media_assets 增加 folder_id，彻底消灭 `unfiled not like '%/%'`
- 如需模糊搜索：加 pg_trgm，并把搜索下推到 DB

4) 结构性体验升级
- admin 列表页 prerender（避免 SSR 冷启动）
- 最终把 admin 做成 SPA（避免每次整页 reload、可做数据缓存）
