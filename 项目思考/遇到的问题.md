# 1. 后台模块字段缺失（例如 FAQ 缺少 items 的 question/answer）

## 现象
- 后台编辑器中，FAQ 模块只显示了 `title` 输入框
- 预期应包含 `items[]`，并可编辑 `question` / `answer`
- 其他模块也可能出现类似情况：例如 services.items、certification.items、equipment.items 等数组字段

## 原因（根因）
后台表单渲染器 [SchemaForm.tsx](file:///Users/javen/Desktop/Javen%20Project/PCB/apps/ems-site/src/components/admin/SchemaForm.tsx) 在渲染对象 schema 时做了过滤：
- 过滤掉了 key 为 `items` 的字段
- 但在我们的内容 schema 设计里，`items` 是一个真实业务字段名（大量模块用它表示列表）

因此任何对象 schema 下名为 `items` 的字段都会被跳过，导致“列表字段整体缺失”。

## 修复
- SchemaForm 对对象 schema 不再过滤 `items`（也不做 `type/items` 的通用过滤）
- 使得 `faq.items[]` 以及其它 `*.items[]` 都能正常渲染出来并可编辑

## 验证方法（手动）
1) 打开 `/ems/admin/pages/ems/`
2) 切到 FAQ Tab，展开 FAQ 模块
3) 确认出现 `Items` 区块，能新增条目，并编辑 `Question` / `Answer`
4) 保存后刷新页面，字段不丢失

## 经验结论（下次规避）
- schema 里的字段名 `items` 非“保留字”，不能在通用渲染层做硬编码过滤
- 如果需要表达数组 schema 的元信息，建议使用不会与业务字段冲突的命名（例如 `__type` / `__items`），或者用判定逻辑区分“数组 schema wrapper”与“业务字段 items”

# 2. 后台 items[] 条目过长，编辑体验冗长（FAQ/Industry 等）

## 现象
- `items[]` 这种 array-of-object 在后台会把每个条目的所有字段完整展开
- 当条目数达到 6+（例如 FAQ 问答列表）时页面会非常冗长
- 更极端的是 Industry 的单个 item 字段层级深、内容长，“一条就一屏”，导致滚动与定位成本很高

## 原因（根因）
- 通用表单渲染器将 array-of-object 以“全部展开表单”的方式渲染
- 没有将条目列表与条目详情分层（缺少 Repeater 的折叠/摘要能力）
- 长文本（answer/paragraph/description 等）默认使用大 textarea，进一步放大纵向占用

## 解决方案（本次采用：A + D）
不改数据结构、不做 Drawer/Modal、不拆表，仅优化交互呈现：

- 方案 A：所有 `items[]` 默认折叠为摘要行，点击“编辑/展开”后才显示完整字段
- 方案 D：长文本字段默认显示 2–3 行预览，点击编辑后展开为 textarea

### 摘要生成规则（智能）
- FAQ：使用 `question`
- Industry：使用 `tab.name + card.title`
- 其它列表：优先 `title` / `name`，否则 fallback 为 “条目 N”

### 保留能力
- 上移 / 下移 / 删除 / 添加条目 均保留

## 验证方法（手动）
1) 打开 `/ems/admin/pages/ems/`
2) FAQ Tab → 展开 FAQ 模块
3) 确认 `items[]` 以摘要行显示，点击条目才展开 `question/answer`
4) `answer` 默认是 2–3 行预览，可点击编辑展开并可收起
5) Industry 模块同理：摘要为 `tab.name + card.title`，展开后显示完整字段

## 经验结论（下次规避）
- ACF Repeater 的关键不是“字段能编辑”，而是“列表信息密度 + 条目详情的分层”
- 对 array-of-object 统一采用：默认折叠 + 摘要行 + 长文本预览，能显著降低滚动成本
- 当单条目复杂度过高（如 Industry），再考虑单模块升级到 Master–Detail；不要一开始就上可视化 builder 或拆表
