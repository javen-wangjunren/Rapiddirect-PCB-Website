# EMS 子站项目介绍（rapiddirect.com/ems/*）

## 1. 背景与目标

公司主站 `rapiddirect.com` 当前基于 WordPress 部署在 Cloudways。随着 EMS/PCB 业务页面增多，继续在 WordPress 内开发会遇到性能与可维护性问题（页面复杂、模块多、交互多、内容迭代频繁）。

本项目目标是在主域路径下构建一个 **独立技术栈的 EMS 子站**：

- 对外正式 URL 保持在主域下，承接主站 SEO 权重
- 前端用 Astro + React + Tailwind 独立开发，避免 WordPress 性能瓶颈
- 数据层使用 Supabase，承载页面内容与 SEO 元信息（以及后续表单/媒体）
- 通过 Cloudflare 按路径分流实现“用户无感知”的无缝衔接

## 2. 正式 URL 规划（强制规范）

子站所有页面必须位于 `/ems/` 下，并统一尾斜杠。

```
/ems/
/ems/pcb-manufacturing/
/ems/pcb-assembly/
/ems/pcb-design/
/ems/components-sourcing/
```

禁止出现：

- 无尾斜杠版本：`/ems/xxx`
- 非 `/ems/` 前缀：`/pcb-assembly/`
- 页面/资源中暴露临时域名（如 `pcb.rapiddirect.com`）

## 3. 整体接入方式（路径分流 + 反向代理）

目标效果：用户访问 `rapiddirect.com/ems/...`，实际页面由新应用输出，但用户侧域名与路径不变。

```
用户访问
rapiddirect.com/ems/*

      │
      ▼
Cloudflare（按 /ems/* 路径分流）
      │
      ▼
Cloudways 新 Application（Astro + React + Tailwind）
      │
      ├─ 页面 HTML
      ├─ /ems/_astro/* 静态资源
      └─ 表单 / 数据请求（Supabase）
```

## 4. 技术栈与职责边界

### 前端（Astro + React + Tailwind）

- Astro：路由、页面渲染、静态构建、SEO 输出（title/description/canonical/sitemap）
- React：仅用于局部交互（如 FAQ 折叠、表单、多步骤交互）
- Tailwind：样式体系与响应式实现

原则：能静态就静态，SEO 主体内容必须直接输出 HTML。

### 数据层（Supabase）

- 承载页面内容（结构化 JSON）
- 承载 SEO 元信息（每页独立设置）
- 承载媒体资源（Storage bucket，如 `pcb-assets`）
- 承载表单数据（后续阶段）

安全要求：

- 前端仅使用 public anon key
- 禁止暴露 service role key
- 通过 RLS 与 CORS 将访问限制在主域（`https://rapiddirect.com`）

## 5. 内容管理策略（Template-driven）

本项目采用“模板驱动”而不是“自由拼装式 page builder”：

- 页面由 `template_type` 决定固定的模块结构
- 页面内容存储在 `page_content.content_json` 中
- 允许部分模块引用全局内容（`global_sections`），但整体结构不自由拖拽

建议数据模型（摘要）：

- `pages`：slug、title、template_type、status
- `page_content`：每页一个 `content_json`
- `seo_meta`：每页一份 SEO 配置（canonical、OG 等）
- `global_sections`：全局复用模块内容（如全局 CTA、认证墙等）

第一阶段先完成 EMS 首页（通常对应 `/ems/`），视觉严格对齐设计稿；交互在实现后结合实际体验再迭代。

## 7. SEO 要求（核心）

Google 收录目标：`https://rapiddirect.com/ems/...`

必须做到：

- canonical：逐页输出，且指向主域正式 URL
- 临时域名（如用于内部测试）必须 `robots.txt` 全站禁止抓取
- sitemap、导航链接、面包屑、Open Graph URL 等全部输出主域正式路径

Astro 配置固定为：

```js
export default defineConfig({
  site: 'https://rapiddirect.com',
  base: '/ems',
  trailingSlash: 'always',
  build: { format: 'directory' }
})
```

## 8. 静态资源与缓存策略

资源路径要求：所有资源必须兼容子路径部署，尤其是构建产物必须在 `/ems/_astro/*` 下可正确访问。

Cloudflare 缓存建议按类型区分：

- HTML 页面：短缓存或绕过缓存（保证内容更新可见）
- `/ems/_astro/*` hash 静态资源：长缓存（例如 1 年）
- API / 表单请求：禁止缓存

## 9. 阶段性交付（MVP）

MVP 验收目标：

- `/ems/` 等核心页面实现并与设计稿对齐
- URL/尾斜杠/资源路径完全符合 `/ems/` 规范
- 每页独立 SEO（title/description/canonical/OG）可配置
- 页面内容可由 Supabase 驱动（后续逐步替换静态写死内容）
