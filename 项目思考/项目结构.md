# EMS 子站项目结构（推荐）

本文档定义本项目的目录结构与职责边界，用于后续：Figma 还原页面、模块化拆分、接入 Supabase、部署到主域 `/ems/` 子路径并满足 SEO 约束。

## 1. 设计目标

- 前端独立技术栈：Astro + React + Tailwind
- 子路径部署：所有页面位于 `/ems/*`，并统一尾斜杠
- Template-driven：页面结构由模板决定，内容来自数据层
- SEO-first：canonical/sitemap/OG 等集中管理，避免散落
- 可维护：模块化 section 组件，支持复用与迭代

# 各层职责说明

&#x20;

## 1. `pages/`

只做路由入口，不堆复杂 UI。

例如 `pcb-assembly.astro` 里只负责：

- 根据 slug 获取数据
- 获取 SEO
- 组合本页面需要的模块

类似：

```
---
const page = await getPageBySlug('/ems/pcb-assembly/')
const content = await getPageContent(page.id)
---

<EmsLayout seo={page.seo}>
  <Hero data={content.hero} />
  <Intro data={content.intro} />
  <Process data={content.process} />
  <Capabilities data={content.capabilities} />
  <Faq data={content.faq} />
</EmsLayout>
```

***

## 2. `components/common/`

放通用模块，不跟某个页面强绑定。

适合放：

- &#x20;Header / Footer&#x20;
- &#x20;Breadcrumb&#x20;
- &#x20;Section Title&#x20;
- &#x20;FAQ Item&#x20;
- &#x20;CTA&#x20;
- &#x20;表单基础字段&#x20;
- &#x20;数据卡片&#x20;

***

## 3. `components/页面名/`

放页面专属模块。\
&#x20;这是你当前最适合的组织方式。

优点：

- &#x20;找文件快&#x20;
- &#x20;跟你 WP 建站习惯一致&#x20;
- &#x20;不会把所有模块全塞进一个 `sections/`&#x20;
- &#x20;第一阶段最清楚&#x20;

***

## 4. `templates/`

现在先留空。\
&#x20;未来如果你发现多个页面结构开始重复，再启用。

比如以后真的出现：

- &#x20;service-template&#x20;
- &#x20;process-template&#x20;

再抽也不迟。

***

## 5. `content/schemas/`

这里不是页面组件，而是**字段结构定义**。\
&#x20;相当于你以前 ACF 里的字段设计文档代码化。

例如：

```
export const pcbAssemblySchema = {
  hero: {
    title: 'string',
    description: 'string',
    image: 'string'
  },
  process: {
    title: 'string',
    steps: 'array'
  }
}
```

***

## 6 `content/defaults/`

默认内容，用来：

- &#x20;本地开发先跑起来&#x20;
- &#x20;Supabase 暂时没数据时 fallback&#x20;
- &#x20;避免空数据导致页面崩掉&#x20;

***

## 7. `lib/supabase/`

放数据查询，不要在组件里直接写 Supabase 查询。

例如：

- `client.ts`：初始化&#x20;
- `queries.ts`：查 pages / seo\_meta / page\_content&#x20;
- `mappers.ts`：把数据库数据转成前端需要的格式&#x20;

***

## 8 `types/`

强烈建议加。\
&#x20;因为后面你一定会有：

- `Page`&#x20;
- `SeoMeta`&#x20;
- `PageContent`&#x20;
- `GlobalSection`&#x20;

早点加会省很多事。

## 9. 资源与图片放置策略（适配 /ems 子路径）

- 优先放 `src/assets/`：走 Astro 构建处理，更不容易出现子路径下资源 404
- `public/` 仅放极少量“固定且明确”的资源（例如 favicon），并谨慎使用根路径引用
